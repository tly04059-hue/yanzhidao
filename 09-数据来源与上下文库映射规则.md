## 一、文档目的

回答三个问题：

1. **小程序的数据从哪里来？**
2. **上下文库的数据怎么变成小程序能用的数据？**
3. **上下文库更新了，小程序数据怎么知道要不要刷新？**

不替代 PRD（PRD 决定要什么数据），不替代数据字典（数据字典定义字段），本文档**只管"来源 → 派生 → 同步"的链路**。

## 三、派生原则

### 3.1 单一可信源

每个数据点必须能追溯到上下文库的**具体文档 + 具体段落**。不允许"凭顾问经验"或"产品设计"作为孤立来源——这些必须先沉淀到上下文库（M3-M9 系列 / 用户洞察 / 行业研究 / 价值观层），再派生到派生层。

**元规则约束**（2026-05-18 新增 · 来自 M7 §七 R10/R11/R12 + 06 v0.2 §四 ER1/ER2/ER3）：派生过程中凡涉及"主力客群措辞 / 政策口径 / 客户尊重"等元规则范围内容，必须先校 06 v0.2 §四 元规则段（ER1 不打"主力客群"标签 / ER2 政策可能性和方向口径 / ER3 尊重三个硬约束），再派生到派生文件。元规则违反 → 派生回炉重写。

**隐私脱敏强制约束**（2026-05-20 新增 · 来自 业务必读 §5.1 圈子重合保护 + 价值观层 §3 信任公式）：任何含**学员真实数据**的派生 JSON（cases.json / students_map.json / peer_profile_cache.json / path_options.json / rules.json）**必须先过《数据脱敏规则.md》全部检查项**才能上线。详见独立文档：`docs/数据脱敏规则.md`（4 层框架：字段脱敏 + 隐私门槛 N ≥ 3 + 反向识别防护 ≤ 4 维 + ww 抽样审核）。脱敏违规 → 派生 JSON 立即下线，不进入运行时。

**脱敏入口位置后移**（2026-05-27 新增 · ww 钦定 · 2026-05-28 按 `docs/数据通路.md` 三阶段收口）：脱敏 4 层框架内容**不变**，仅入口位置移动：
- **旧规则**：派生 JSON 生成时跑 check_privacy
- **新规则**：派生到选校工具消费层（`data/derived-from-context/*.json`）前一步由 `derive_*.py` 脚本自动完成脱敏 + `check_privacy.py` 校验
- **M8 SSOT 内部不脱敏**：M8-学员案例.csv 升级为「全量内部 SSOT」（v0.3 起含真实姓名 / 工作单位全称 / 详细地址 / 地级市 / 民族州 / 数据状态 共 6 个内部字段），仅在派生到选校工具消费层时脱敏

### 3.2 派生方法分类（4 类 → 5 类 → 6 类）

| 方法 | 含义 | 示例 |
|---|---|---|
| **映射** | 字段一对一对应 | M9 §19 川渝 MPA 表 → `path_options.json` |
| **总结** | 提取核心结论 | M6 隐性规则 11 条 → `rules.json` 中的部分规则 |
| **提炼** | 从大段叙述抽出结构化要点 | 认知框架 §6.5 MPA 5 门核心课 → `understand_long_form.json` §2 段 |
| **归纳** | 跨多个文档归纳出共性 | M8 案例库 + 用户洞察访谈 → `cases.json` |
| **对话协议**（V0.2 新增） | 把多步反向追问对话流派生为前端表单/后端状态机 | M7 §七 R13 / 06 v0.2 DP001 英数短板反向追问 → `dialog_protocols.json`（三层追问 → questions.json 子集 + answer_option.json 子集 + 跳转规则）|
| **压缩派生**（V0.3 新增） | 跨多源文档汇总后，强压缩到「一句话规则」+ 留长版参考路径 | 专项/02 v1.0 §三「按系统-财政」5 段详情 + M5 触发场景 + M7 §七 + M9 院校库 + ww 校对 → `understand_long_form.json` §3 一句话规则「财政 78% 选党校经济学（业务对口）」|
| **脚本派生**（V0.6 新增） | 通过 Python 脚本对 M8 SSOT 做算法聚合（含三维分桶 / fallback_chain 派生 / 半自动候选派生 / V6 cases 派生 / 脱敏出口校验等 · 详见 `docs/数据通路.md` 阶段 2/3）| M8 csv + 5 张 .xlsx 报名表 → `students_map.json` `stats_by_match_key` 段（`scripts/derive_students_map.py`）；投稿叙事 + M8 扩展字段 → `cases.json` V6 投稿 27 条（`scripts/derive_v6_consumer_cases.py`）；M8/上岸汇总 v04 69 条 → V6 schema 并入（`scripts/derive_v04_v6_cases.py`）；正式 `cases.json` 当前 125 条 |

派生方法必须在派生文件的 frontmatter 里标注。

**压缩派生说明**（V0.3 新增 · 处理 v3.3 §3 4 维度 19 条规则模式）：

PRD v3.3 §4.2 §3「不同情况下规则不一样」段，把原 v3.1 的 4 个维度详情页（每页 5 段 ≈ 20 段内容）压缩成 19 条一句话规则。这种"多源 → 一句话"派生不属于现有 5 类——它本质是「跨源归纳 + 强压缩 + 留长版参考路径」三步合一。

**压缩派生的 4 步流程**：

1. **多源汇总**：列出所有源文档 + 具体段（M2 / M3 / M5 / M7 / M9 / 价值观层 / 专项/02 v1.0 / ww 历次校对）
2. **核心结论提取**：每个维度子项提取 1 个核心结论（如"财政"维度提取"业务对口度高 + 78% 选党校经济学"）
3. **一句话压缩**：≤ 30 字 · 含「数据 / 排序 / 原因」中至少 1 项 · 严格遵守 ER1/ER2
4. **留长版参考**：派生文件 `_meta.long_version_ref` 字段指向原长版（如 `专项/02 §六.1`）便于反查

**为什么单列一类而不是归到提炼+归纳**：

- 提炼是「单源 → 多要点」；归纳是「多源 → 多要点」；**压缩派生是「多源 → 一句话 + 留反查路径」**——三者输入输出维度都不同
- 压缩派生需要"对账日志"（19 条规则每条对应的源是什么、是否和源一致），单列一类才能要求每条规则在派生时记录这个日志
- 06 推荐规则表对账时，可凭压缩派生的对账日志快速判断"19 条规则 vs 06 §五 13 条规则"的对应关系

**对话协议派生说明**（V0.2 新增）：

对话协议（dialog protocol）规则（如 06 v0.2 DP001）不能用前 4 种方法派生——前 4 种都是单步输出，对话协议是多步对话流。对话协议派生时：

1. 把每层追问问题派生为 `questions.json` 的子集（如 DP001 第一层"是真实测试过还是恐惧" → 1 个 question）
2. 把每层追问选项派生为 `answer_option.json` 的子集（如 DP001 第一层选项 → 测试过 / 没测试 2 个 answer_option）
3. 把层间跳转规则派生为 `dialog_protocols.json`（核心载体）
4. 把每层信息透出（如 DP001 第一层 B 分支"非全管综英语二满分 100 / 过线 34-38 分"）派生为 `copy.json` 子集
5. 把最终路径建议跳转派生为常规推荐流（接 P001/P002/P003）

**dialog_protocols.json 派生结构示例**：

```json
{
  "_meta": { /* 同 4.1 */ },
  "data": [
    {
      "protocol_id": "DP001",
      "trigger_keywords": ["英语不行", "数学不行", "怕英数", "英语过不了线"],
      "layers": [
        {
          "layer_id": "L1",
          "name": "能力真伪辨",
          "question_ref": "Q-DP001-L1",
          "branches": [

    }

}
```

### 3.3 派生层独立性

派生层文件**不引用上下文库的文件路径或动态读取**。派生时已把内容完整提炼到派生文件里。

理由：
- 服务器迁移时，派生文件直接打包上传，不需要带上下文库
- 上下文库改动**不会自动改派生层**，需要走"检测 → 评估 → 重新派生"流程，保证可控

### 3.4 院校库特例

**院校库的数据由服务器单独维护，不走本文档定义的派生链路**：

- 院校字段会按业务调整
- 院校数据量大、更新频率高、可能涉及外部采集
- 服务器侧有独立的"院校数据维护"流程（参考已删除的旧 Agent 设计的部分思路）

## 四、派生文件格式规范

每个派生文件（json / md / yaml）**必须**有 frontmatter / metadata 头，格式如下：

### 4.1 JSON 派生文件

```json
{
  "_meta": {
    "derived_object": "questions",
    "source": [
      {
        "path": "02-市场与用户/用户洞察/M4-年龄段学历场景矩阵.md",
        "version_date": "2026-05-11",
        "section": "全文",
        "method": "提炼"
      },
      {
        "path": "02-市场与用户/用户洞察/M5-触发场景与决策情境.md",
        "version_date": "2026-05-11",
        "section": "§2 四类触发信号",
        "method": "提炼"
      }
    ],
    "derived_at": "2026-05-15",
    "derived_by": "Claude + ww 确认",
    "sync_status": "synced",
    "next_review_at": "2026-08-15"
  },
  "data": [
    /* 真正的数据内容 */
  ]
}
```

### 4.2 Markdown 派生文件

```markdown

  - path: 02-市场与用户/院校库/院校库认知框架.md

（真正内容）
```

### 4.3 必填字段说明

| 字段 | 含义 | 必填 |
|---|---|---|
| `derived_object` | 派生对象类型（对应 04 数据字典中的对象） | 是 |
| `source[].path` | 上下文库源文件路径 | 是 |
| `source[].version_date` | 源文件的修改日期（用于比对） | 是 |
| `source[].section` | 引用了源文件的哪一段（精确到节 / 表 / 行） | 是 |
| `source[].method` | 派生方法：映射 / 总结 / 提炼 / 归纳 | 是 |
| `derived_at` | 本派生文件生成时间 | 是 |
| `derived_by` | 派生执行者 | 是 |
| `sync_status` | 同步状态：synced / stale / pending-review | 是 |
| `next_review_at` | 下次复查时间（季度复查） | 否 |

## 五、状态同步机制

### 5.1 sync_status 状态机

| 状态 | 含义 | 触发条件 |
|---|---|---|
| `synced` | 派生文件与上下文库源一致 | 派生完成且最近一次检测无差异 |
| `stale` | 上下文库源已更新，派生文件未跟新 | 检测脚本发现 `source[].version_date < 源文件最新修改日期` |
| `pending-review` | ww 暂未确认是否需要刷新 | 检测出 stale 后 ww 暂未决定 |

### 5.2 同步检测时机

| 时机 | 谁触发 | 动作 |
|---|---|---|
| 上下文库重要文档更新后 | ww 主动提示 / Claude 在编辑时主动检测 | 跑 `_sync_state.json` 检测，标 stale |
| 季度复查 | 每季首周 | 全量检查所有派生文件的 `next_review_at` |
| 数据上传服务器前 | 上传脚本预检 | 不允许 `stale` 状态的文件上传 |

### 5.3 _sync_state.json 总表

派生层根目录维护一份总表，方便扫描：

```json
{
  "last_check_at": "2026-05-15",
  "derived_files": [
    {
      "file": "questions.json",
      "sync_status": "synced",
      "last_sync_at": "2026-05-15",
      "sources_checked": 2
    },
    {
      "file": "rules.json",
      "sync_status": "stale",
      "last_sync_at": "2026-04-20",
      "stale_reason": "M9-院校库分析.md 已更新（2026-05-14 > 2026-04-20）"
    }
  ]
}
```

### 5.4 何时"刷新"

派生文件的更新不是自动的。流程如下：

```
1. 检测器发现 stale → 标记
2. Claude / ww 评估：
   - 上下文库的改动是否影响这个派生文件？
   - 是的话，重新派生（更新内容 + 版本号 + sync_status=synced）
   - 否的话，仅更新 source[].version_date，不动内容（保留 synced）
3. ww 审核派生结果
4. 上传服务器（或者本地保留，等下次批量上传）
```

## 七、迁移到服务器的规则

### 7.1 上传时机

- 数据对象首次完成派生（含 ww 审核）
- 派生文件从 stale 重新变 synced 后
- 院校库特例由服务器侧另行处理

### 7.2 上传内容

- 派生文件 `data/derived-from-context/*.json / *.md`
- **不上传** frontmatter / `_meta`（这是本地工程治理用的，上服务器后保留在仓库里，运行时数据库不需要）
- 服务器接收脚本负责把 json 数据导入对应数据表

### 7.3 上传后版本对账

每次上传后，记录"上传到服务器的派生文件版本"和派生层本地版本，确保一致。可放在 `_sync_state.json` 里加一个 `last_uploaded_at` 字段。

## 十、维护原则（v0.2 升级）

1. **PRD 改 → 重新评估数据对象清单 → 刷新映射表 → 重新派生**

3. **派生文件改 → 必须更新 frontmatter 的 `derived_at` 和 `sync_status`**
4. **永远不允许"绕过上下文库直接改派生文件"**——除非这个数据本来就是小程序专属的（如界面文案 / 交互参数）
5. **小程序专属数据**单独标注 `derived_object: ui_only`，不强制上下文库来源

7. **06 v0.2 是 rules.json 的执行链权威**（v0.2 新增）：派生 rules.json 时主要从 06 v0.2 §五取，不从 M7 §七取；M7 §七 仅作为 evidence_source 字段的来源标注

## 十.bis、派生约束信息的可见性约束（v0.3 新增 · 2026-05-19）

> **核心声明**：派生约束信息 = 后台数据治理元数据，**不对外展示**。用户在小程序里看到的只是派生数据的渲染结果，不会看到"派生自哪个 L1 文档 / 版本日期 / meta_rule_compliance"等元数据。

### 10.bis.1 不可见范围（绝不暴露给用户）

- `_meta.source[].path`（派生自哪个上下文库文档）
- `_meta.derived_at` / `_meta.derived_by` / `_meta.next_review_at`
- `_meta.meta_rule_compliance`（ER1/ER2/ER3 校验状态）
- `_meta.sync_status`（synced / stale / draft）
- `data/derived-from-context/_sync_state.json` 全文件
- HTML 节点上的 `data-derived-from="..."` 注释属性
- 03 内容审核清单 / 06 推荐规则表 / 09 派生映射规则等内部治理文档

### 10.bis.2 可见范围（用户能看到的）

- 派生数据 `data` 段渲染后的内容（题目 / 选项 / 结果文案 / 学校卡片 / 故事卡 / 政策卡 / 判断书）
- 派生数据的"用户可读来源说明"——只来自 §五数据来源页（如"过考率口径：研知道辅导班 1,000+ 同学样本"）

### 10.bis.3 治理用途（仅给 ww + 派生执行人）

派生约束元数据的三个治理用途：

1. **追溯**：某条派生数据出错时，凭 `_meta.source[].path` 反查 L1 文档原文
2. **重派生触发**：L1 文档更新时，扫 `derived_at` < L1 `version_date` 的派生文件，列入重派清单
3. **月度对账**（v3.2 PRD §7.1 反哺规则）：ww 拉报表时，凭 `experiment_id` / `meta_rule_compliance` 关联推荐规则迭代

### 10.bis.4 前端 HTML 标注规范（v3.2 PRD § 配合）

4.0 原型在关键数据驱动节点（题目 / 结果模块 / 推荐 / 故事卡 / 政策卡 / 判断书 / 同样画像同学）加 `data-derived-from` 属性，标明派生自哪个 JSON。**该属性供后台审计 / 灰度对账 / CI 校验消费，不在 UI 上展示**。

示例：

```html
<section class="screen" id="screen-poster" data-derived-from="data/derived-from-context/_archived/可删除-share_assets.json">
  <!-- 内容 ... -->
</section>
```

CI 校验脚本（待开发）每次构建时扫描所有 `data-derived-from` 属性，检查：

- 对应 JSON 文件存在
- JSON 的 `_meta.sync_status === "synced"`
- JSON 的 `_meta.meta_rule_compliance` 全部通过
