# 数据脱敏规则

> 本文档是研知道选校诊断小程序**派生层 JSON 的强制硬规则**。任何含学员真实数据的派生 JSON 在上线前必须过本文档全部检查项。
>
> **强制性**：违反本规则的派生 JSON 一律不能上线服务器，不能被小程序运行时消费。
>
> **适用范围**：所有从 M8 学员案例 / M9 院校库 / 报名表 / 上岸统计 / 访谈记录 派生出来的 JSON。
>
> **决策日期**：2026-05-20 ww 拍板，参见会话决策（信任公式落地 + 业务必读 §5.1 圈子重合保护）。

---

## 一、为什么必须脱敏

研知道的用户群体有 3 个特殊性，决定了隐私保护必须比一般 ToC 产品**严**：

1. **体制内人群** — 个人信息泄露后果不可逆（影响仕途、组织考察、人际关系）
2. **圈子重合**（业务必读 §5.1）— 同事 = 同学 = 邻居，识别一个人很容易，3 个维度组合往往就能定位
3. **信任承诺**（价值观层 §3）— 研知道公开承诺"不卖鸡汤、不出卖用户"，隐私失误直接破坏信任公式（自我导向感↑ + 可靠度↓）

**结论**：脱敏不是"合规义务"，是"产品立场"。

## 二、涉及隐私的派生 JSON 清单（5 个）

| JSON | 隐私敏感度 | 数据来源 | 主要风险 |
|---|---|---|---|
| **cases.json** | 🔴 最高 | M8 学员案例 1,123 条 | 姓名/单位/年龄/学历/报考结果，反向识别风险 |
| **students_map.json** | 🟡 中 | M8 1,592 报名样本 + 地址 | 城市分布、系统类型，少数民族地区数据量少易识别 |
| **peer_profile_cache.json** | 🔴 最高 | K-NN 实时匹配 | 多维聚合时可能反向定位 |
| **path_options.json** | 🟢 低 | 路径选择聚合 | 已是聚合数据，但仍需检查门槛 |
| **rules.json** | 🟢 低 | 历史样本归纳 | 已是抽象规则，但 evidence 引用需脱敏 |

**不涉及隐私的派生 JSON**（无需走本规则）：
`copy.json` / `dialog_protocols.json` / `meta_rules.json` / `questions.json` / `understand_long_form.json` / `policy_basis.json` / `about.json` / `usage_guide.json` / `pass_rate.json`（聚合统计）/ `schools_meta.json`（院校属性，无个人数据）

## 三、4 层隐私保护框架

### 第 1 层 · 字段级脱敏（派生时强制）

任何派生 JSON 中包含学员个人字段，**必须**按下表脱敏：

| 字段类型              | 原始数据示例            | 脱敏规则                   | 派生 JSON 中的值                |
| ----------------- | ----------------- | ---------------------- | -------------------------- |
| **姓名**            | 王俊明               | "X 同学" + 仅保留姓          | "王同学"                      |
| **单位**            | 成都市青羊区税务局第三分局     | 系统类型 + 子类（**不到具体单位**）  | "区县税务局"                    |
| **地区**            | 成都市青羊区光华大道        | **地级市层级**（不到区县/街道）     | "成都"                       |
| **年龄（默认）**     | 33 岁              | **5 岁年龄段**（不写具体年龄）     | "30-35 岁段"                 |
| **年龄（v6 卡片例外 · 2026-05-28 1C 钦定）** | 33 岁 | **模糊词 + 具体岁数**（"约 N 岁" / "N 岁左右"） | "约 33 岁" |
| **学历**            | 四川大学计算机科学与技术全日制本科 | 类型（**不到学校 / 专业**）      | "全日制本科"                    |
| **党员身份**          | 2015 年入党          | 是 / 否（**不写党龄 / 入党时间**） | "党员"                       |
| **报考结果**          | 四川省委党校经济学 2024 届  | 路径名（**不到具体院校 / 届数**）   | "党校在职"                     |
| **联系方式**          | 手机号 / 微信号 / 邮箱    | **完全不出现**在派生 JSON 中    | —                          |
| **身份证 / 工号 / 学号** | 任何编号              | **完全不出现**              | —                          |
| **访谈原话**          | 任何含识别性的具体细节       | 保留原话但去除可识别细节           | "我同事去年也考了党校" → "我身边有人考了党校" |

**强制规则**：派生时如发现某字段无法按上表脱敏（如年龄已是关键画像维度无法替换），必须**直接删除该字段**，不允许部分脱敏。

**1C 模糊化例外（2026-05-28 ww 钦定）** · 适用 cases.json V6 消费层的 `age_concrete` 字段：

V6 #14 案例卡片要求展示具体岁数以提升信息密度，与默认"5 岁年龄段"冲突。ww 钦定折中方案 **1C 模糊化**：

| 字段 | 派生原值 | 1C 模糊化输出 | 适用场景 |
|---|---|---|---|
| `age_band`（cases.json 基线字段）| 33 → "30-35 岁段" | 仍是"30-35 岁段" | #22 chip 筛选 / 桶聚合 |
| `age_concrete`（v0.3 新增字段）| 33 | **"约 33 岁"** | #14 案例卡 / #29 长图卡片标题 |

**1C 模糊化规则**：
- 派生格式：`f"约 {N} 岁"` · 不允许 `"33 岁"` 这种精确格式
- 派生脚本：`derive_v6_consumer_cases.py` 已实现（2026-05-28）
- 反向识别风险：模糊词降低画像精度（±2 岁波动语义），与"5 岁段"等价的隐私强度
- 限定使用：仅 `age_concrete` 字段 · `age_band` 仍走默认 5 岁段规则
- 不可扩展：除 v0.3 cases.json 已批准外，其他派生 JSON 不允许使用此例外

### 第 2 层 · 隐私门槛（聚合时强制）

任何"用学员数据做聚合展示"的场景，必须满足下表的样本量门槛：

| 场景 | 门槛 | 触发时降级处理 |
|---|---|---|
| **K-NN 同画像匹配**（§4.18 / §4.16 §6）| **N ≥ 3** | 显示"样本不足，建议 1 对 1 顾问诊断" |
| **单位类型聚合** | 每类 ≥ 3 人才独立展示 | 合并到上级（如"区县税务" → "区县财税"）|
| **地区聚合**（学员地图 §4.19）| 每地市 ≥ 3 人才独立展示 | 合并到省（如"果洛州 2 人" → 不展示 或合并到"民族地区"）|
| **案例库展示**（§4.11）| 每个画像组合 ≥ 3 个独立样本支撑 | 不展示该画像案例 |
| **过考率细分** | 每个细分维度 ≥ 30 人才展示分项数据 | 仅展示总体过考率 |

**Why N ≥ 3**（2026-05-20 ww 校对调整 · 原 N ≥ 5）：
- 低于 3 时容易反向识别（4 维画像 + 同地市 + 同系统 + N≤2 → 圈内人能猜出是谁）
- N=3 是隐私保护与可用性的平衡点 — 5 太严会导致大量画像组降级为"样本不足"
- **代价**：4 维画像组合 + N=3 时仍有反向识别风险（圈子重合保护要靠第 3 层"≤4 维"铁律 + ww 抽样审核兜底）
- **要求**：N=3 时**额外加强第 3 层维度组合检查**，确保 3 维以下展示组合

**Why 案例库仍 N ≥ 3**：案例库是叙事化展示，每个画像至少要有 3 个独立样本才能保证 "X 同学" 的故事不会指向唯一一个真人。与 K-NN 匹配阈值一致。

---

### 第 3 层 · 反向识别防护（最关键）

**铁律**：永远不能在**同一展示位**组合 4 个以上脱敏维度。

**原因**：
- 即使每个字段都按第 1 层脱敏到位
- 5 个维度组合（地级市 + 单位子类 + 年龄段 + 党员 + 学历）
- 数据库里可能就 1 个人能匹配
- → 圈内人能反向识别

**执行规则**：

| 展示场景 | 允许的最大维度组合数 | 必须砍掉至少 1 个维度 |
|---|---|---|
| 案例卡（§4.11 / §4.16 §6 故事卡）| **最多 4 维** | 5 维中至少去 1 个（如要么不写地级市，要么不写单位子类）|
| 同画像同学（§4.18 / §4.16 §6）| 5 维都参与匹配，但展示时只显示**聚合维度**（不展示个体）| 展示"9 位选党校 / 3 位选 MPA" 而不是"王同学/李同学..."|
| 学员地图（§4.19）| 仅展示 **2 维**：地级市 + 学员数 | 不混合年龄/系统 |
| 城市详情面板 | 地级市 + 业务系统类型（最多 3 个），**不显示个体** | 不写"该市有 X 位 30 岁党员选党校" |

**小众组合预警**：
- 民族地区（甘孜/凉山/阿坝）涉及具体年龄 + 单位时**强制额外脱敏**（合并年龄段到 10 岁，单位合并到"乡镇级"）
- 任何展示组合，全数据库筛查后样本 < 3 → 触发降级（不展示 或合并维度）

### 第 3 层 · 3.4 合成案例脱敏（v1.2 新增 · 2026-05-27 ww 钦定）

某些画像（profile_combo_id）真实样本数 N<3 时，允许通过合成案例补足到 3 条，让画像组对外可展示。**合成必须严守 4 条铁律 + 渲染强制脱敏**。

**触发条件**：

- 某 PC 真实样本 N<3（cases.json 案例库展示门槛 · §三 第 2 层）
- 真实样本 N≥1（N=0 不合成 · 走 fallback_chain L3 "1v1 引导"）

**合成 4 条铁律**：

| 铁律 | 规则 |
|---|---|
| **铁律 1 · 字段来源真实** | 每个字段必须来自真实样本（primary + borrowed）· 绝不凭空生成 |
| **铁律 2 · 显式标注** | 合成案例必须含 `_is_synthesized: true` + `_composition_note` + tags 含 "合成画像" |
| **铁律 3 · ≤4 维展示** | 卡片渲染时强制省略 `region` 字段（与 §三 第 3 层"≤4 维"铁律一致 · 防反人肉）|
| **铁律 4 · key_quote 不嫁接** | `key_quote` 必须与 `display_name` 同源（即 primary 学员）· 绝不能给合成"虚拟同学"安别人的话 |

**字段拼接表**（派生脚本 derive_synthesized_cases.py 实施）：

| 字段 | 来源 |
|---|---|
| `case_id` | 新生成 `SYN_{primary_case_id}_{seq}` |
| `display_name` | primary（真实学员标识 · 配回 key_quote）|
| `system` | primary（强绑）|
| `outcome` | primary（强绑）|
| `key_quote` / `key_quote_source` | primary（铁律 4）|
| `region` | borrowed（**渲染时强制省略**）|
| `age_band` | borrowed |
| `chosen_path` | primary（同 path 借鉴）|
| `chosen_school` | borrowed |
| `_is_synthesized` | true |
| `_synthesized_from` | { primary, borrowed[] } |
| `_composition_note` | 自动生成说明 |

**反向识别检查（派生时强制）**：

- borrowed 提供的 `region + age_band` 组合在全库真人 ≥3 → 借鉴细节不孤
- primary 的 `system` 在全库真人 ≥1 → primary 本身合规

**渲染层脱敏要求（小程序同事消费时强制）**：

- 卡片标 "合成画像" badge（小红色高亮）
- **不展示 `region` 字段**
- 显示 `_composition_note` 在卡片底部（小灰字）
- 不展示 `_synthesized_from` 细节（仅内部追溯）

**违规处理**：

- 合成案例未显式标注 _is_synthesized=true → 立即下线
- 合成案例 key_quote 嫁接（与 display_name 不同源）→ 立即下线
- 合成案例展示 region → 立即下线

### 第 4 层 · 审核机制

| 时机 | 谁做 | 怎么做 |
|---|---|---|
| **派生 JSON 生成时** | 自动化脚本 | 跑第 1 层（字段脱敏完整性）+ 第 2 层（隐私门槛）+ 第 3 层（维度组合数）检查，任何失败直接拒绝输出 |
| **JSON 上线前** | **ww 抽样审核** | **至少 10% 样本** 人工审核 + **至少 1 个民族地区案例** + **至少 1 个小众组合** + **至少 1 个 K-NN 匹配场景** |
| **每季度** | ww | 复审已上线 JSON，新增样本是否符合规则；ww 圈层内是否出现"被识别" 投诉 |
| **用户投诉时** | ww | 立即下线相关 JSON → 排查脱敏漏洞 → 重新脱敏 → ww 复核 → 重新上线 |

## 四、违规处理

| 违规类型 | 处理 |
|---|---|
| 派生 JSON 含未脱敏字段（如真名 / 联系方式 / 具体单位）| **立即下线 JSON** → 删除已上线版本 → 修复后重新上线 → ww 复核 |
| 派生 JSON 违反隐私门槛（N < 3 仍展示）| **立即下线** → 修复降级逻辑 → ww 复核 |
| 派生 JSON 违反维度组合铁律（>4 维 / 民族地区未额外脱敏）| **立即下线** → 砍维度 → ww 复核 |
| 圈层内"被识别"投诉 | **立即下线相关 JSON** → 排查根因 → 修复并加强脱敏 → 通知投诉人 → ww 公开说明 |

## 五、自动化检查脚本规约

派生 JSON 生成流程必须集成以下自动化检查（建议作为 09 派生层 Pre-commit hook）：

```
派生 JSON 生成
    ↓
[Check 1] 禁字段扫描
   - grep 真名 / 手机号 / 微信号 / 身份证号 / 具体单位名 / 具体院校名 / 具体年龄
   - 失败 → 拒绝输出
    ↓
[Check 2] 脱敏字段格式校验
   - 姓名必须是 "X 同学" 格式（仅姓 + 同学）
   - 地区必须是地级市层级（不含"区""县""街道"）
   - 年龄必须是 "X-Y 岁段" 格式
   - 失败 → 拒绝输出
    ↓
[Check 3] 隐私门槛检查
   - 每个聚合组合的样本量 ≥ 对应门槛
   - 失败 → 拒绝输出
    ↓
[Check 4] 维度组合数检查
   - 每个展示项的维度数 ≤ 4
   - 失败 → 拒绝输出
    ↓
[Check 5] 小众组合预警
   - 民族地区样本 + 年龄/单位组合的二次脱敏检查
   - 失败 → 触发额外脱敏
    ↓
通过 → 标记 "脱敏审核通过 · YYYY-MM-DD" → 等 ww 抽样审核 → 上线
```

**实现位置**：脚本放在 `研知道/03-产品与服务/选校工具/scripts/check_privacy.py`（待建）
