## 一、技术决策总结

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 前端框架 | uni-app | 一套代码编译小程序 + H5 + 未来 App |
| 后端框架 | Node.js (Express) | 已有代码骨架，生态成熟 |
| 数据库 | SQLite（V1）→ MySQL（按需升级） | V1 流量小够用，容器内运行简单 |
| 部署平台 | 微信云托管 | 免备案（小程序）、零运维、Docker 部署 |
| AI 推荐 | RAG + 多模型降级链 | 政策知识库检索 + LLM 生成，豆包/DeepSeek 为主，Claude 预生成内容 |
| H5 定位 | **全功能等同小程序** | H5 是通用入口，短信/公众号/各场景都能打开 |
| H5 托管 | Cloudflare Pages | 免费 CDN、部署快 |
| 智能跳转 | H5 检测微信环境 → 引导打开小程序 | 用户不熟练手机操作，自动选最优路径 |
| DNS | Cloudflare | 免费 DNS 管理 |
| 域名 | V1 需备案（H5 公网访问） | 小程序不依赖备案可先上 |

### 智能环境检测与跳转

```javascript
// H5 入口页检测逻辑
function detectAndRoute() {
  const ua = navigator.userAgent.toLowerCase();
  const isWechat = ua.includes('micromessenger');

  if (isWechat) {
    // 在微信内 → 提供"打开小程序"按钮（同时 H5 也可用）
    showMiniProgramBanner();
    // 使用微信 URL Scheme 跳转小程序
    // wx.miniProgram.navigateTo 或 URL Scheme
  }
  // 无论什么环境，H5 完整运行
}
```

**设计原则：H5 永远可用，微信内额外提供小程序入口。** 不要强制跳转，因为用户可能不习惯操作。显示一个温和的横幅："在微信小程序中体验更好 → 打开"，用户可以忽略继续用 H5。

**H5 版本需要一个已备案域名**（如 `api.yanzhidao.com`），指向微信云托管的公网地址。小程序版本不需要。

## 三、AI 推荐模型（三层架构 + 多模型降级）

### 三层架构

```
第一层：规则引擎（确定性，不用 AI）
  engine.js → 决定推荐什么学校、什么专业、触发哪些警告
  → 100% 可审计、可追溯，绝不用 AI 做推荐决策

第二层：政策知识库 + RAG（合规性）
  从政策原文库中检索与用户画像相关的政策段落
  → AI 说的话有出处，不会编造政策

第三层：LLM 文本生成（人味）
  把规则结果 + 检索到的政策 + 案例素材 → 写成像真人顾问的话
  → 只负责"说话方式"，不负责"说什么"
```

### 为什么需要三层

| 场景 | 纯规则引擎 | +RAG+LLM |
|------|----------|-----------|
| 固定路径匹配（系统→专业→院校） | 够用 | 不需要 |
| **个性化推荐理由** | 只能模板拼接 | 像真人顾问根据你的情况在说话 |
| **"这条路径对你意味着什么"** | 固定 3 句话 | 根据系统+岗位+目标动态组织政策解释 |
| **案例匹配叙述** | 固定引语 | 更自然的"这个人和你很像"叙述 |
| **了解 Tab 18 个详情页** | 全靠人工写 | Claude 预生成初稿，ww 审核定稿 |
| **边界场景** | 可能遗漏 | 兜底处理未覆盖的组合 |

### 模型选择：多模型分工

| 用途 | 模型 | 理由 |
|------|------|------|
| **内容预生成**（浏览详情页、模板话术库） | Claude Sonnet | 一次性写完，质量最高，不计成本 |
| **实时推荐理由生成** | 豆包（字节）或 DeepSeek | 便宜（Claude 的 1/10）；原生理解体制内语境 |
| **降级兜底** | 无需模型 | 人工审核过的固定模板话术 |

豆包/DeepSeek 原生理解"竞争上岗""遴选""职级并行"，不需要额外解释。

### RAG 检索方案

```
政策知识库（本地文件，V1 不需要向量数据库）
├── 职级并行规定.txt
├── 四川遴选公告2025.txt
├── 重庆遴选公告2025.txt
├── 公务员调任规定.txt
├── 职称评审改革意见.txt
├── 党校学历认可文件.txt
├── 组织考察办法.txt
└── ...

检索逻辑（简单关键词匹配）：
  用户画像 {系统:财政, 目标:遴选, 地区:四川}
  → 匹配关键词 → 取出相关政策段落
  → 喂给 LLM 作为上下文
```

V1 政策文件不超过 20 个，简单关键词匹配足够，不需要向量数据库。

### 实时推荐流程

```
用户答题完成
     ↓
Express API 收到全部答案
     ↓
Step 1: 规则引擎（engine.js）
  → 首选/备选路径 + 排除规则 + 案例匹配
  → 输出结构化 JSON
     ↓
Step 2: 检查缓存
  → 用户画像 hash → 查 ai_cache 表
  → 命中 → 直接返回（不调 AI）
  → 未命中 → 继续
     ↓
Step 3: RAG 检索
  → 根据系统+目标+地区检索政策段落
     ↓
Step 4: LLM 生成（豆包/DeepSeek）
  → 输入：规则结果 + 政策段落 + 用户画像 + 语气示例
  → 输出：推荐理由 + 路径解释 + 案例叙述
  → 写入 ai_cache
     ↓
合并输出 → 返回前端
```

### Prompt 设计（语气控制）

```
你是研知道的资深选校顾问，帮体制内用户解读选校方案。

语气要求：
- 像一个懂行的学长在跟你聊，不是客服在念稿
- 直接说结论，再解释为什么
- 用"你"不用"您"
- 可以说"说白了""其实""坦白讲"
- 不说"加油""相信自己""未来可期"
- 不用"亲""呢""哦"
- 每段 100 字以内

好的示例：
"你在财政局，经济学专业跟你日常业务直接对口。
 说白了，竞争上岗的时候，评委看你专业方向，
 经济学比政治学更能让人觉得'这人是对口的'。"

不好的示例（绝对不要）：
"经济学专业非常适合您！相信通过努力学习，
 您一定能在未来的职业道路上取得更大的成就！"
```

### 降级策略

```
豆包/DeepSeek 正常 → 返回 AI 个性化理由
     ↓ 失败
尝试备用模型（DeepSeek ↔ 豆包互为备用）
     ↓ 失败
降级到人工审核过的模板话术（engine.js 内置）
     ↓
规则引擎独立可跑，永不依赖 AI
```

FROM node:20-alpine

   - `DEEPSEEK_API_KEY`（DeepSeek 备用模型）

## 七、安全防护

| 层 | 措施 |
|----|------|
| API 签名 | 请求头带时间戳 + HMAC 签名，防止 API 被直接调用 |
| 频率限制 | 同一 openid/IP 10 分钟内最多 3 次推荐请求 |
| 推荐逻辑保护 | 决策树和规则参数在后端，前端只发答案、收结果 |
| 内容保护 | 案例和浏览内容通过 API 动态加载，不硬编码 |
| AI 调用保护 | Claude API Key 在环境变量，不暴露；AI 缓存避免重复调用 |
| 数据隔离 | SQLite 文件在容器内，不对外暴露 |

## 十、技术风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 豆包/DeepSeek API 不稳定 | 推荐理由生成失败 | 双模型互为备用 → 模板话术兜底；热门组合预缓存 |
| AI 编造政策 | 合规风险 | RAG 架构：AI 只基于检索到的政策原文生成，不让它"知道"政策 |
| 微信云托管冷启动 | 首次访问慢 | 配置最小实例数 = 1（常驻） |
| SQLite 并发限制 | 高流量时写入冲突 | V1 流量不构成问题；后续可迁移 MySQL |
| H5 备案延迟 | H5 版本延后上线 | 小程序先上，H5 等备案 |
| uni-app 跨端兼容问题 | 小程序和 H5 表现不一致 | 开发阶段双端同步预览测试 |
| H5 在某些场景无法跳转小程序 | 用户体验割裂 | H5 永远完整可用，不强制跳转；微信内显示温和引导横幅 |
