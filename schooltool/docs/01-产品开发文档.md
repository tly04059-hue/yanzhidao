## 一、产品定位

### 这个产品是什么

**面向川渝体制内用户的学历路径决策产品。**

不是"帮你选四川党校的工具"，而是帮用户完成从"我要不要读研"→"我为什么要读"→"我该走哪条路"→"下一步做什么"的完整判断链条。

"选校"只是其中一个环节，不是全部。

### 产品的三层能力

| 层   | 能力                  | 承载形态           |
| --- | ------------------- | -------------- |
| 解释层 | 公开信息结构化，讲清学历作用和路径差异 | "了解"Tab（浏览式）   |
| 匹配层 | 根据个人画像推荐具体方案        | "测一测"Tab（答题式）  |
| 承接层 | 导流到顾问、引入备考服务        | 首页入口 + 结果页 CTA |

### 一句话定义

川渝体制内人的在职读研路径决策工具 — 公开透明的政策解读、按你情况匹配的真实案例、3 分钟给出个性化选校方案。

### 产品形态

微信小程序。原因：

- 微信分享卡片原生支持，体制内人工作群分享点击率远高于链接
- 服务通知和订阅消息可推送
- 微信授权登录、自动出现在历史列表
- 体制内人网络环境不稳定（尤其民族地区），小程序有缓存秒开

## 二、目标用户

### 核心人群（按读研意向阶段分）

| 阶段 | 用户心理 | 产品接入方式 |
|------|---------|------------|
| 犹豫期 | "我是不是也该读个研？" | "了解"Tab：浏览政策、案例，理解学历作用 |
| 决策期 | "我该考党校还是 MPA？" | "测一测"Tab：答题得出个性化方案 |
| 行动期 | "我已经决定了，怎么备考？" | 首页入口 → 备考方案、资料、联系顾问 |

**三个阶段不是强制顺序，产品三个入口并列存在，用户可以从任何阶段进入。**

### 核心人群（按画像分）

| 类型 | 特征 | 核心诉求 |
|------|------|---------|
| 补短板型（~48%） | 非全日制本科，30-42 岁 | 低成本拿到研究生学历，不被学历卡住 |
| 巩固型（~52%） | 全日制本科，28-40 岁 | 增强晋升竞争力，让优势更明显 |
| 难度规避型 | 想读研但怕统考 | 党校作为可行替代，了解差异 |
| 模糊焦虑型 | "是不是也该读个研" | 需要政策解释和案例帮他理清 |

### 当前不覆盖

- 非川渝地区用户
- 以学术深造、全日制脱产为目标的人
- 只关注全国院校排名、不关心体制内适配逻辑的人

## 三、核心价值

### 对用户

- 模糊期：看政策 + 看案例，理解学历到底有什么作用（不是鸡汤，是政策依据）
- 决策期：3 分钟搞清楚自己适合读什么，看到和自己一样的人怎么选的
- 行动期：得到清晰的下一步动作 — 备考方案 / 资料 / 顾问

### 对研知道

- 每个访客自动生成完整画像（系统/地区/目标/预算/年龄/学历/党员）
- 顾问拿到的不是一个微信号，而是一份诊断卡
- 降低重复咨询成本，把重复解释变成标准化产品
- 复合型引流产品：低门槛进入，浏览式留存 + 答题式转化
- 品牌信任：过考率公开透明 + 政策依据明确 + 不卖课卖判断

### 核心差异

市面上没有同类产品。现有咨询成本高的根源不是信息太少，而是信息分散、判断标准不统一、解释重复发生。这个产品把研知道的研究资产（政策研究 + 人事制度 + 院校数据 + 学员案例）公开透明地结构化，用户能自己看明白，不只靠顾问解释。

## 四、开发阶段规划

### Phase 1：产品定义与结构验证（当前阶段）

**目标：** 想清楚、画清楚、能体验

- 收敛产品定位文档
- 确定页面结构和流程
- 做出可交互原型
- 确定视觉风格

**交付物：**
- 产品开发文档（本文件）
- V1 PRD
- 交互原型（HTML）

**验证标准：**
- 用户能快速理解产品是干什么的
- 能顺利走完一条完整路径
- 结果页让用户觉得"有价值"

**不做：** 后端、数据库、管理后台

### Phase 2：内容与规则沉淀

**目标：** 确保数据准确、逻辑正确

- 院校数据逐项审核
- 案例库 20-25 条就绪（ww 提供素材）
- 推荐规则全组合验证
- 所有文案审核

**交付物：**
- 院校数据集（JSON）
- 案例库（JSON）
- 题目配置（JSON）
- 内容审核通过记录

**验证标准：**
- 所有院校数据 ww 确认
- 16 条逻辑校验清单全部通过
- 案例可公开、无隐私风险

### Phase 3：技术实现

**目标：** 把产品做出来

- 技术方案定稿
- 前端开发（小程序所有页面）
- 后端开发（推荐引擎 API）
- 前后端联调

**交付物：**
- 可运行的小程序
- 后端 API 服务
- 数据库

### Phase 4：测试与上线

**目标：** 确认正确、部署发布

- 全流程端到端测试
- 真机 iOS/Android 测试
- 部署到阿里云
- 微信小程序提审发布

### Phase 5：运营与迭代

**目标：** 推广、监测、优化

- 渠道推广（公众号、朋友圈、顾问 1v1）
- 数据监控（漏斗、转化率、画像分布）
- 持续优化（A/B 测试、案例补充、话术调整）

## 五、当前版本边界

### V1 三 Tab 结构

```
[ 首页 ]   [ 了解 ]   [ 测一测 ]
```

### V1 先做

**首页（总入口）**
- 三个主动作入口：了解 / 测一测 / 备考
- 信任数据锚（过考率 + 学员数）
- 其他入口：院校查询 / 联系顾问 / 关于研知道

**"了解"Tab（浏览式 — 模糊期用户）**
- 四种浏览入口：按系统 / 按目标 / 按学历 / 按年龄
- 每个浏览详情页：政策依据 + 实际作用 + 相似案例 + 推荐路径
- 路径对比页（党校 vs MPA vs MBA）
- 院校库浏览
- 案例库浏览

**"测一测"Tab（答题式 — 决策期用户）**
- 6-8 道题的答题流程
- 加载过渡页
- 个性化结果页（首选 + 备选 + 路径意义 + 案例 + 信任数据 + CTA）

**备考内容（从首页入口进）**
- 备考方案（党校 / 管综）
- 免费资料领取
- 加入备考群
- 课程服务介绍

**其他**
- 留资功能（结果页 + 独立领取入口）
- 分享海报

### V1 不做

- 复杂账号体系和登录
- 数据管理后台（Phase 2 再做）
- 精细的推荐算法（决策树足够）
- 全国范围扩展
- 社区 / 学习管理功能
- 分享裂变机制（先做基础分享）
- 独立的备考 Tab（备考入口从首页进，不做成一级导航）

## 六、验证框架

每个阶段进入下一阶段前，必须满足对应验证标准。

| 阶段 | 关键验证问题 | 验证方式 |
|------|------------|---------|
| Phase 1 | 流程和信息层级对不对？ | ww 手机上走完原型 |
| Phase 2 | 数据和逻辑对不对？ | 逻辑校验清单 + ww 逐项确认 |
| Phase 3 | 产品能正常运行吗？ | 全流程端到端测试 |
| Phase 4 | 线上环境一切正常吗？ | 真机测试 + 灰度发布 |
| Phase 5 | 用户觉得有价值吗？ | 转化率 + 用户反馈 |

## 八、核心原则

1. **先想清楚，再动手写代码。** Phase 1 不碰技术实现。
2. **用户体验 > 功能完整。** 3 步内完成任何目标。
3. **数据准确 > 数据丰富。** 宁可少做院校，不能错。
4. **推荐逻辑是核心资产。** 全部在后端，前端不碰。
5. **所有院校数据和案例需 ww 确认后才能上线。**
6. **字要大、按钮要大。** 主要用户 35 岁+。
7. **调性要正经。** 体制内人分享到工作群不能觉得 low。
