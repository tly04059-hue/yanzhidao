## 一、文档目的

本文件是推荐系统的**工程实现层文档**。

它不替代推荐逻辑说明文档，而是把"解释性内容"进一步拆成可落地的规则单元，供：

- 前端联调结果页
- 后端实现规则引擎
- 内容侧校对推荐理由
- 顾问回查特殊情况
- **新增**：跨环境迁移（自包含确保规则可独立部署）

**与 M7-选校判断逻辑.md 的边界**：

- **M7 §七**：规则名 + 一句话规则本质 + 价值观/事实/经验级依据 + 与其他模块的关系（**回答"为什么这样判"**）
- **本文档**：完整规则本体 + 工程化 condition + 实现逻辑 + 输出文案 + 兜底 + 状态（**回答"怎么实现这个判"**）
- 两边规则名一致（如 P001 体制内党员→路径 A 优先），但内容互不重复——独立读懂、独立维护
- M7 是依据链权威，本文档是执行链权威

## 三、规则类型（5 类 → 6 类）

### 3.1 路径优先规则（P）

用于判断首推路径。前缀 `P`。

### 3.2 排除规则（E）

用于把不适用路径排除。前缀 `E`。

### 3.3 阻断规则（B）

用于直接结束常规推荐，改为承接路径。前缀 `B`。

### 3.4 风险提示规则（R）

用于不改变主结果，但追加提示。前缀 `R`。

### 3.5 备选路径规则（A）

用于结果页生成第二、第三推荐路径。前缀 `A`。

### 3.6 应答框架规则（DP · V2 新增）

用于客户主诉触发反向追问，对话跑完才给路径建议。前缀 `DP`。与前 5 类的区别——前 5 类直接输出结果，DP 类先启动 dialog protocol。

## 四、元规则段（V2 新增 · 3 条执行级强约束）

**所有规则在 condition 命中后，output_copy 必须先满足以下元规则**：

### ER1 不打"主力客群"标签

- **约束**：output_copy 不得含"主力客群 / 最强 / 最集中 / 谁占多数 / 主要面向"等暗示概率分布的措辞
- **判断依据**：主力客群描述以"客户管理客户资料的清单分布"为依据，不靠抽检/AI 推导/惯性判断
- **校验时机**：output_copy 文案审核阶段（人工 / 自动）
- **违反处理**：阻止规则上线，回炉重写
- **依据**：经营战略总纲 §5.2 公理第 ⑥ 类 / M4 §八 不做主力判断 / 院校库认知框架 §十.5 注 / M7 §七 R10

### ER2 政策口径

- **约束**：output_copy 涉及政策结论必须用"基于核实政策事实的可能性和判断方向"措辞，**不做确定的因果论断**
- **可用措辞**："政策支持的可能性" / "判断方向" / "可能的应用场景" / "对应的政策依据"
- **禁用措辞**："一定会" / "肯定能" / "必然导致" / "保证升职/调任/落编"
- **校验时机**：output_copy 文案审核阶段
- **依据**：上下文价值观层 §3.8 C2 / M9 §5.1·补 / §4.1·补 / M7 §七 R11

### ER3 尊重三个硬约束

- **约束**：output_copy + 流程交互必须满足：
  - 不催单（禁用"今天最后一天 / 名额仅剩 / 立即报名"）
  - 不要求客户公开渠道传播（禁用"分享朋友圈 / 转发到群"）
  - 不让客户朋友圈被同事看到尴尬（私域承接为主，公域裂变慎用）
- **三个硬约束来源**：党纪政纪 / 不犯错导向 / 圈子重合（生活圈=工作圈）
- **校验时机**：output_copy + 交互流程审核阶段
- **依据**：M5 §五.2 三个硬约束 / 业务必读 §5.1 / 内容生产手册 §一.3 合规三线 / M7 §七 R12

### 4.5.2 使用规则

1. **规则 output_copy 引用政策时**：直接写"基于 [PD-XX] 政策事实的可能性和判断方向"，**不重复列政策名/文号**（已在本表汇总）

3. **政策更新触发**：本表 PD-XX 任一行内容更新 → 关联规则的 output_copy 必须重新校验 → 若涉及结论变化（如年龄红线调整），关联规则升 v0.5

    - 分支 B：不刚（仅内部竞争防御）→ 党校"不考英数"才是真省力点 → 推 P001
    - 分支 C：**先有证再优化**（防御性立场：先确保不掉队，再考虑加分）→ 推 P001 党校优先（性价比+速度+不考英数）+ P002 备选（如后续遴选/调任需求出现）
  - **第三层 · 能力偏好辨**（v0.3 校准：原"应试型/表达型"→"理科思维/文科思维/说不准"，用户语言级校准）：问"你更倾向哪种思维方式？"

- **output_action**:
  - **不直接给路径**
  - 启动三层反向追问对话流（每层一个问题 + 选项）
  - 三层全部跑完后才进入 P001/P002/P003 推荐

| **总分** | 100 + 20 | 5 维度基础权重 100，非党员场景加成最高 +20 |

#### K-NN 匹配阈值

- **完全相似（≥85）**：5 维全部一致 OR 4 维一致 + 第 5 维相近 → 直接展示完整数据
- **强相似（70-85）**：3-4 维一致 → 展示但提示"匹配度参考"
- **弱相似（55-70）**：2-3 维一致 → 不展示，进 ER3 隐私门槛 N<3 同等处理（建议直接咨询顾问）
- **不相似（<55）**：不进 K-NN 候选

#### ER3 隐私门槛配套

- 相似度 ≥85 + N ≥ 3（同画像样本数）→ 展示完整数据
- 相似度 ≥85 但 N < 3 → 触发 ER3 不展示
- 相似度 <85 → 不展示

| 维度 | 细化取值范围 | 数据源 |
|---|---|---|
| 体制类型 | 8 类（同上）| 客户答题 Q1 + Q2 |
| 目的 | 6 类（同上）| 客户答题 Q6 / 主诉 |
| 学历 | 4 类（同上）| 客户答题 Q3 |
| 地区 | 7 类（同上）| 客户答题 Q1 + 推断 |

| 党员身份 | 党员 / 非党员 / 预备党员 | 客户答题 Q4 |

```

```

## 九、规则冲突裁决机制（V2 新增）

### 9.1 类型优先级（高 → 低）

1. **ER 元规则**（约束所有规则的 output_copy）
2. **B 阻断规则**（直接结束推荐）
3. **E 排除规则**（剪掉路径）
4. **DP 应答框架规则**（不直接给路径，先反向追问）
5. **P 路径优先规则**（推路径）
6. **R 风险提示规则**（不改主结果，加提示）
7. **A 备选路径规则**（生成第二、第三推荐）

### 9.2 维度优先级（解决同类型规则冲突）

- **目的级 > 身份级 > 能力级**
- 例：体制内党员（身份级 P001）+ 目的=遴选（目的级 P002）→ 优先 P002 双证，但 P001 作为"备选路径"加入

## 十一、与上下文库的对齐声明

### 11.1 自包含原则

- **运行时**：所有规则的 condition / output_action / output_copy / fallback 完整在本文档内；选校工具迁移到其他环境（服务器 / 第三方 / 跨部门）时，仅需带本文档 + `04-数据字典.md`，不依赖上下文库实时读取
- **维护时**：`evidence_source` 字段标了上下文库源头（M7 §七 R[N] / L1 文档段落），但**不是运行时依赖**——是规则维护时的对账接口

### 11.2 与 M7 的依赖关系

- **M7 §七**：规则名 + 一句话规则本质 + 价值观/事实/经验级依据来源（**依据链权威**）
- **本文档**：完整规则本体 + 工程化条件 + 实现逻辑 + 输出文案 + 兜底 + 状态（**执行链权威**）
- 两边规则名一致（如 P001 / E001 / B001），但内容互不重复——独立读懂、独立维护

### 11.3 对账机制

- M7 §七 更新时，本文档对应 rule_id 必须**同步检测**（手动或自动）
- 本文档更新时，**不强制反向同步 M7**（因为本文档是执行细化，M7 是依据层）
- 但元规则 ER1/ER2/ER3 的变化必须双向同步（元规则是价值观传导）

### 11.4 跨环境迁移清单

选校工具迁移时携带：

- `06-推荐逻辑规则表.md`（本文档）
