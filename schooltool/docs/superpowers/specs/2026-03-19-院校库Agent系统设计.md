# 研知道·院校库Agent系统设计

> 版本：v1.1 | 日期：2026-03-19
> 定位：MBA/MPA全国院校数据基础设施
> 上层应用：选校诊断小程序、企微机器人、内部运营

---

## 一、系统定位

院校库是**底层数据资产**，支撑三个出口：
1. **小程序**（面向川渝考生）：院校查询、选校诊断的数据源
2. **企微机器人**（面向内部+考生）：即时查询、审核通知、调剂信息推送
3. **Web管理后台**（面向内部）：数据审核、数据源配置、异常处理

数据挖掘全国，价值聚焦川渝。

### 2.2 数据库表结构

**核心设计原则：一校一项目一记录**

同一所学校的MBA全日制、MBA非全日制、MPA是**不同记录**（学费、招生人数、分数线完全不同）。

```sql
-- 院校主表（学校级别，去重）
universities (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  code            VARCHAR(10) UNIQUE,    -- 研招网学校代码，唯一标识
  name            VARCHAR(100),          -- 四川大学
  province        VARCHAR(20),           -- 四川
  city            VARCHAR(20),           -- 成都
  tier            ENUM('985','211','双一流','普通'),
  is_self_marking BOOLEAN DEFAULT FALSE, -- 是否34所自划线
  created_at      DATETIME,
  updated_at      DATETIME
)

-- 项目表（一校一项目一记录，核心表）
programs (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  university_id   INT REFERENCES universities(id),
  type            ENUM('MBA','MPA'),
  study_mode      ENUM('全日制','非全日制'),
  department      VARCHAR(100),          -- 商学院/公共管理学院
  year            INT,                   -- 招生年份（如2026）

  -- P0 核心决策字段
  tuition_total   DECIMAL(10,2),         -- 学费总额
  tuition_yearly  DECIMAL(10,2),         -- 学费每年
  duration        VARCHAR(20),           -- 学制：3年
  enrollment      INT,                   -- 招生人数
  requirements    TEXT,                  -- 报考条件

  -- P1 重要参考字段
  schedule        VARCHAR(100),          -- 上课方式：周末/集中/晚班
  campus          VARCHAR(100),          -- 校区/上课地点
  directions      JSON,                  -- 培养方向
  certifications  JSON,                  -- AACSB/EQUIS/AMBA/CAMEA
  discipline_rank VARCHAR(10),           -- 学科评估等级
  accepts_transfer BOOLEAN,              -- 是否接受调剂
  has_early_interview BOOLEAN,           -- 是否有提前面试
  contract_type   ENUM('定向','非定向','均可'),
  scholarship     TEXT,                  -- 奖学金信息

  -- P2 深度字段
  retest_subjects TEXT,                  -- 复试科目
  retest_books    TEXT,                  -- 复试参考书目
  retest_weight   VARCHAR(20),           -- 初试/复试权重比
  retest_ratio    VARCHAR(20),           -- 复试差额比
  employment_url  VARCHAR(500),          -- 就业报告链接
  alumni_note     TEXT,                  -- 校友网络评价

  -- 元数据
  data_completeness DECIMAL(3,2),        -- 数据完整度评分（0-1）
  last_verified_at  DATETIME,            -- 上次人工校验时间
  created_at      DATETIME,
  updated_at      DATETIME,

  UNIQUE(university_id, type, study_mode, year)
)

-- 分数线表（独立存储，支持多年历史）
score_lines (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  program_id      INT REFERENCES programs(id),
  year            INT,                   -- 2025
  total_score     INT,                   -- 总分线 151
  english_score   INT,                   -- 英语线 35
  composite_score INT,                   -- 综合线 70
  source_id       INT REFERENCES data_sources(id),
  created_at      DATETIME
)

-- 历年录取数据（独立存储，支持趋势分析）
enrollment_history (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  program_id      INT REFERENCES programs(id),
  year            INT,
  applicants      INT,                   -- 报考人数（稀缺）
  admitted        INT,                   -- 录取人数
  retest_count    INT,                   -- 进复试人数
  ratio           DECIMAL(5,2),          -- 报录比（计算字段）
  source_id       INT REFERENCES data_sources(id),
  created_at      DATETIME
)

-- 字段变更历史（追踪数据变化）
field_history (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  program_id      INT REFERENCES programs(id),
  field_name      VARCHAR(50),           -- 变更的字段名
  old_value       TEXT,                  -- 旧值
  new_value       TEXT,                  -- 新值
  changed_at      DATETIME,
  source_id       INT REFERENCES data_sources(id),
  approved_by     VARCHAR(50)            -- 审核人（"auto"或"ww"）
)
```

**数据完整度评分计算**：
- P0字段每填1个得6分（13个×6=78分）
- P1字段每填1个得3分（10个×3=30分）
- P2字段每填1个得1分（10个×1=10分）
- 满分118分，归一化为0-1

### 2.3 数据源模型

```
data_sources（数据源配置表）
├── id
├── name: "院校官网" / "MBAChina" / "研招网" / "希赛网" ...
├── type: "official" / "aggregator" / "ugc"
├── weight: 1-5（你配置的权重）
├── auto_approve_threshold: 权重≥N时自动入库
├── url_pattern: 采集URL模板
└── status: active / disabled

data_records（采集记录表）
├── id
├── school_id
├── source_id
├── field_name: "tuition" / "enrollment" / ...
├── field_value: 具体值
├── fetched_at: 采集时间
├── source_url: 数据来源URL
├── status: "pending" / "approved" / "rejected" / "auto_approved"
└── conflict: bool（是否与其他源冲突）
```

### 2.3 审核规则引擎

```
审核决策流程：

1. Agent采集到某校某字段的新值
2. 查该字段在其他数据源的值
3. 判断：
   ├── 多源一致（≥2个权重≥3的源一致）→ 自动入库
   ├── 仅单一高权重源（权重≥4）→ 自动入库
   ├── 源间不一致 → 标记冲突 → 进"待审核"队列 → 通知你
   ├── 仅低权重源（权重≤2）→ 必须人工确认
   └── 字段值异常（学费为0、招生人数>1000等）→ 标记异常 → 待审核
```

## 四、数据采集策略

### 4.1 初始化流程（一次性）

```
Step 1: MBAChina全量批量采集
├── 遍历34省MBA院校列表 → 约300所
├── 遍历34省MPA院校列表 → 约250所
├── 逐校抓取详情页 → 解析__NUXT__ JSON
├── 入库为"基准数据"（来源=MBAChina，权重3）
└── 预计耗时：2-3小时（限速避免被封）

Step 2: 川渝+云贵重点院校深度补全（约25所）
├── 逐校访问官网招生简章
├── 提取P0+P1+P2全量字段
├── 与MBAChina数据交叉比对
├── 冲突项进审核队列
└── 你逐条确认

Step 3: 研招网校验
├── 验证院校清单完整性
├── 补充考试科目、研究方向
└── 修正异常数据
```

### 4.2 持续更新策略

| 时间 | 触发方式 | 动作 |
|------|---------|------|
| 3-4月 | 自动 | 分数线专项采集（国家线+各校复试线） |
| 5-6月 | 自动 | 提面批次采集（MBA为主） |
| 9-11月 | 自动 | 全量刷新（新一届招生简章发布） |
| 3-4月 | 自动 | 调剂信息采集（调剂系统3月底开放） |
| 随时 | 手动 | 你说"更新XX大学"，Agent即时跑 |

### 4.3 变更检测

每次采集后自动与库中数据比对：
- 学费变了 → 高优先级通知（考生最关心）
- 招生人数变了 → 通知
- 新增/取消提面 → 高优先级通知
- 新增院校开始招MBA/MPA → 通知

## 五、数据源权重配置

文件：`config/sources.yaml`（你来编辑）

```yaml
sources:
  - name: 院校官网
    type: official
    weight: 5
    auto_approve: true  # 权重5的硬数据自动入库
    note: 最权威，招生简章原文

  - name: 研招网
    type: official
    weight: 4
    auto_approve: true  # 硬数据自动入库
    note: 官方招生目录

  - name: MBAChina
    type: aggregator
    weight: 3
    auto_approve: false  # 需与其他源一致才自动入库
    note: 数据最全，批量初始化首选

  - name: 希赛网
    type: aggregator
    weight: 2
    auto_approve: false
    note: 分数线汇总较好

  - name: 知乎/小红书
    type: ugc
    weight: 1
    auto_approve: false  # 必须人工确认
    note: 面经、报录比等体验型数据
```

### 审核阈值规则

```yaml
review_rules:
  # 硬数据（学费/招生人数/分数线）
  hard_data:
    auto_approve_when: "max_source_weight >= 4 OR (consistent_sources >= 2 AND min_weight >= 3)"
    require_review_when: "conflict OR max_source_weight <= 2"

  # 软数据（上课方式/培养方向/认证）
  soft_data:
    auto_approve_when: "max_source_weight >= 3"
    require_review_when: "conflict"

  # 稀缺数据（报录比/奖学金/面经）
  scarce_data:
    auto_approve_when: "never"  # 始终需要人工确认
    require_review_when: "always"
```

| 页面 | 功能 |

## 八、与小程序的对接

### 8.1 API设计

| 接口 | 方法 | 说明 | 调用方 |
|------|------|------|--------|
| GET /api/schools | GET | 院校列表（支持筛选） | 小程序+后台 |
| GET /api/schools/{id} | GET | 院校详情 | 小程序+后台 |
| GET /api/schools/compare | GET | 多校对比 | 小程序 |
| GET /api/schools/search | GET | 院校搜索 | 小程序+企微 |
| GET /api/transfer | GET | 调剂院校列表（当季） | 小程序+企微 |
| POST /api/review/approve | POST | 审核通过 | 后台+企微 |
| POST /api/agent/run | POST | 触发采集 | 企微+后台 |

### 8.2 与现有小程序的关系

现有小程序的Node.js后端保持不变，通过HTTP调用院校库的FastAPI接口获取数据。两套服务独立部署，通过API通信。

```
小程序前端 (uni-app)
    ↓
小程序后端 (Node.js Express) —HTTP→ 院校库API (FastAPI)
    ↓                                    ↓
推荐引擎/案例匹配                     MySQL（院校数据）
```

## 十、采集安全与运维

### 10.1 反爬/限流策略

| 数据源 | 请求间隔 | 降级策略 |
|--------|---------|---------|
| MBAChina | 3-5秒/页 | __NUXT__格式变更时自动暂停+通知；备用HTML解析方案 |
| 院校官网 | 5-10秒/页 | 403/超时后自动跳过，记录失败列表，下次重试 |
| 研招网 | 10秒/请求 | API返回异常时切换UA+Cookie；不做大规模并发 |

### 10.2 格式变更监控

MBAChina的`__NUXT__` JSON是最大脆弱点。防御机制：
- 每次采集前先抓1个已知院校，校验返回字段数
- 字段数骤降（<50%正常值）→ 自动暂停 → 企微通知你
- 备用方案：切换为HTML + BeautifulSoup解析

### 10.3 数据安全

- MySQL每日自动备份（mysqldump + 阿里云快照）
- API鉴权：小程序端用API Key，管理端用JWT Token
- 审核API仅企微机器人和Web后台可访问（IP白名单）

### 10.4 错误处理

- 采集失败：自动重试3次（间隔递增），仍失败则记录日志+跳过
- 系统监控：每日采集成功率统计，低于80%时企微告警
- 定时任务：用系统crontab触发（不嵌入FastAPI进程），任务状态持久化到数据库

## 十一、已知局限和应对

| 局限 | 影响 | 应对 |
|------|------|------|
| 报录比大部分院校不公开 | 考生最想知道但无法提供 | 能采到的采，采不到标"暂无"，鼓励学员反馈 |
| 奖学金非全日制普遍无 | 字段空值多 | 如实标注，有值的是加分项 |
| MBAChina数据可能有误 | 个别字段异常（如学费=0） | 异常检测自动标记+人工兜底 |
| 院校官网结构各异 | 无法100%自动化 | 先模板化高频院校，长尾靠MBAChina覆盖 |
| 部分院校SPA渲染 | requests抓不到 | Playwright备用（约10-20%院校需要） |
| 微信公众号难抓取 | 部分提面信息只在公众号发 | 暂不自动化，靠人工补录或搜狗微信搜索 |
