## 一、V1 目标

帮助川渝体制内用户完成学历路径决策的全链路接入 — 从"我要不要读研"到"我该走哪条路"到"下一步做什么"。

### V1 要接住的三类用户

| 用户心理             | 产品入口     | 核心体验           |
| ---------------- | -------- | -------------- |
| 犹豫期："我是不是也该读个研？" | "了解"Tab  | 看政策、看案例，理解学历作用 |
| 决策期："我该考什么？"     | "测一测"Tab | 答题 → 个性化方案     |
| 行动期："我决定了，怎么备考？" | 首页备考入口   | 备考方案、加企微顾问 |

### V1 业务目标

- 复合型低门槛引流产品：浏览式留存 + 答题式转化
- 降低重复咨询成本：顾问拿画像而不是空白微信号
- 建立品牌信任：政策透明 + 过考率公开 + 不卖课卖判断
- **以防御性产品立场承接焦虑**：不承诺升职加薪/影响力变大/改变命运；承诺降低决策成本 + 提升服务体验 + 提高备考效率（业务必读 §5.1 + 价值观层 §3.7 + 商业全景 §0.2 起点承诺三层结构）
- **AI 时代提供确定性资产**：在政策红利 + 末等退出 + 国考限研岗持续抬升的环境下，做一件结果可控、可见、可用的事（商业全景 §2.1 第五股力量）

## 三、设计原则

### 3 步原则

从打开到获取关键信息，控制在 3 步以内。

| 用户目标            | 步骤               | 步数  |
| --------------- | ---------------- | --- |
| 做选校诊断           | 首页/测一测→答题→结果     | 3   |
| 了解本系统学历作用       | 首页→了解→滚到 §3「按系统」 | 2   |
| 路径对比（路径 A 党校 vs 路径 B 统考非全） | 了解→路径对比          | 2   |
| 查院校信息           | 首页→院校库→（可选：点路径节点过滤）→详情 | 3-4（破例：路径决策图是 §4.9 核心新设计，路径节点过滤为可选层） |
| 看过考率怎么算的        | 首页→点信任数据任一卡      | 2   |
| 看研知道同学分布        | 首页→学员地图          | 2   |
| 联系咨询            | 首页→加企微咨询         | 2   |


### 操作深度控制

| 步数    | 原则          | 示例                         |
| ----- | ----------- | -------------------------- |
| 第 1 步 | 一眼看懂，一个动作   | 首页→点"我还在犹豫"                |
| 第 2 步 | 内容铺开，不让选    | 了解 Tab 一份长文滚下去 |
| 第 3 步 | 可以有信息量，辅助判断 | 长文 §3「不同情况下规则不一样」横扫 4 维度   |
| 第 4 步 | 行动收口        | 留资 / 分享 / 测一测              |

**DP 触发路径破例说明**：答题流程命中 DP001（英数短板）等应答框架规则时，需要先跑反向追问再给推荐——此情况答题总步数从 6-8 → 9-11 步。3 步原则只针对非 DP 触发路径。DP 触发时在追问首屏明示"为了给你更准的推荐，再回答 3 个问题（约 1 分钟）"，让用户预期可控。

### 用户特征适配

- 主要用户 35 岁+：字要大（正文 16px+）、按钮要大（96rpx+）
- 体制内人群：稳重、可信、不花哨，政策和数据说话
- 注意力有限：能少就少，干扰最小化
- 95% 从微信打开：移动端优先，PC 不适配
- 具体数值落地（字号 / 按钮 / 间距）见 设计/design.md

### 内容原则

- **不做鸡汤，只做政策解释和案例匹配**
- 所有"为什么读研"的回答必须基于政策依据或学员数据
- 所有推荐必须给出依据（政策文件号 或 学员统计）
- **用防御性表述（保留可能性 / 不掉队 / 占位），不用进攻性承诺（升职加薪 / 影响力变大 / 改变命运）**——参业务必读 §5.1 + 价值观层 §3.7
- **元规则强约束**：所有展示文案必须先通过 ER1（不打"主力客群"标签）+ ER2（政策口径用"可能性和判断方向"，禁用"一定会/必然/保证"）+ ER3（不催单 / 不要求公开传播 / 私域承接为主）三条元规则——详见 06 §四 + 专项/03-内容审核清单.md

### 4.2 "了解"Tab — 长文全景图
**定位：** 一份完整的「川渝体制内学历路径全景」长文。用户不需要做任何选择（不选维度 / 不答题 / 不填信息），从头滚到底就能拿走完整认知和分类规则。

**用户画像：** 目标不明确，或此刻不想做决策——给信息的规则感 + 获得感，不增加任何决策压力。

**和测一测的关系：** 不是漏斗，是**互通通道**。测一测每题旁可一键跳到本页对应章节锚点（见 §4.6）。

### 4.6.1 应答框架（DP）· 反向追问内联流程

当 Q9 命中 `trigger_keywords`（如"英语不好/数学不好"触发 DP001），答题流程暂停常规推荐，进入 inline 反向追问：

```
Q9 提交 → 检测 DP 命中 → 启动 dialog_protocol → 跑完 L1/L2/L3 → 进推荐
```

**DP001 英数短板反向追问的三层结构**（与 06 DP001 dialog_protocol 一致）：

| 层 | 问题 | 选项 | 分支映射 |
|---|---|---|---|
| **L1 能力真伪辨** | "聊路径前先确认下——你说'英语不行/数学不行'，是真实考过试测下来不行，还是没正式测过觉得自己不行？" | A. 测试过 / B. 没测试过 | A → 问具体分数；B → 透出门槛信息"非全管综英语二满分 100 只需过 34-38 分；管综数学是高中+大学初级，不是考研数学一/二难度" |
| **L2 动机需求辨** | "你的目标场景对学位类型有要求吗？" | A. 必须双证（遴选/调任/职称硬要求）/ B. 不刚（仅内部竞争防御）/ C. 先有证再优化 | A → 推路径 B 统考非全（双证）；B → 推路径 A 党校（"不考英数"才是真省力点）；C → 推路径 A 党校优先 + 路径 B 备选（防御性立场：先确保不掉队再考虑加分） |
| **L3 能力偏好辨** | "你更倾向哪种思维方式？" | A. 理科思维 / B. 文科思维 / C. 说不准 | A → 推路径 B 统考非全管综（数学+英二+逻辑靠方法和推导）；B → 推路径 A 党校（全主观题靠材料积累+论述）；C → 进 L3.b 二级追问 |
| **L3.b 努力偏好辨**（仅 L3 选 C 看到） | "你更愿意在理解上努力，还是在大量背诵上努力？" | A. 理解努力 / B. 背诵努力 | A → 类管综 → 路径 B；B → 类党校 → 路径 A |

**未来扩展性**：答题引擎做成"线性主流程 + dialog_protocol 状态机"，dialog_protocols.json 数据驱动（09 派生层已定）。V1 只实装 DP001，未来 DP002/DP003（如"年龄到了"反向追问、"学费贵"反向追问等）只改数据不改代码。

**API 调用**：DP 应答框架**完全在前端状态机执行**——trigger_keywords 检测在前端做，dialog_protocols.json 从派生层静态拉取（前端缓存），DP 中间不调后端。最终 `dialog_history` 数组随 `/api/recommend` 一并提交。

**§2 目录设计要点**：

- **位置**：紧接封面之后，画像摘要之前 — 让用户翻到第二屏就看清全图结构
- **形态**：纵向列表 6 行（不是水平 tab），每行结构：`[01]  段名  ┄┄┄  一句话简介`
- **每行示例**（v0.2 ww 校对：03/04 取消数字，05 改"可以做什么"）：
  - `[01]  我的判断       ┄┄┄  首推路径 + 备选 + 匹配度`
  - `[02]  路径含义       ┄┄┄  这条路对你意味着什么`
  - `[03]  参考政策       ┄┄┄  你该知道的政策`
  - `[04]  相似故事       ┄┄┄  与你画像相似的同学`
  - `[05]  行动清单       ┄┄┄  接下来可以做什么`
  - `[06]  风险与边界     ┄┄┄  我们能给你什么 / 不能给什么`
- **视觉规范**：
  - 序号 `[01]` 用赤土橙小方块 + 白字
  - 段名 serif 16px 加粗
  - 简介 sans 13px 弱色 (text-3)
  - 行间细线 1px dashed border 区隔
  - 不抢主推荐视觉权重 — 整段视觉密度低于 §3 画像摘要
- **小程序内交互**（保存到相册前）：每行**可点击锚点跳转**到对应段
- **保存为长图后**：作为视觉指引（不可点，但用户扫一眼能知道整图有什么 + 大概多长）
- **目录段自身高度**：约 200-280px（不超过首屏一半）

**§6 参考政策 4 张卡内容规则**：

- 4 张卡片**根据用户路径动态选取**（不同路径触发的政策依据不同）
- 默认 4 张（延迟退休 / 国考研岗 / 末等退出 / 三条政策叠加）来自上下文价值观层 + 业务必读 §5 + 商业全景 §2.1
- 每张卡含：政策名 + 一句话解读 + 与用户路径的关联（"为什么这个政策对你的路径有影响"）
- 文案严格遵守 ER2（不用"一定会 / 必然 / 保证"），用"基于核实政策的可能性和方向"措辞
- **正式实现时数据从案例库 + 政策依据库 实时取**，不在 PRD hardcode

**§7 相似故事 4 张卡内容规则**：

- 4 张卡片**根据 K-NN 5 维相似度算法实时匹配**（同 §4.18 同样画像的同学怎么选模块）
- 维度：年龄段 / 体制内层级 / 学历起点 / 地区 / 价值取向
- 匹配阈值：5 维至少 4 维相同
- 隐私门槛：N ≥ 3 才展示，N < 3 时本段降级为"样本不足，建议结合 1 对 1 顾问诊断"
- 每张卡含：脱敏画像 + 实际选择 + 一句话原话（来自访谈样本）+ 上岸状态
- **正式实现时数据从 M8 学员案例库 + 09 派生层 `cases.json` 实时取**，不在 PRD hardcode

**导出形态**：

- 长图 jpg/png（≈ 750×3000px · v0.2 含目录段后高度 +200-280px）
- 一键存到相册（微信小程序 `saveImageToPhotosAlbum` API）
- 不需要登录、不需要关注、不需要分享

**技术实现**：前端用 canvas 或 html2canvas 渲染长图。

**API 调用 / 数据组合**：

- 不新增专门 API — 完全复用 `/api/recommend` 返回 + 客户端模板渲染
- 数据来源：
  - `path_meaning` + `primary` + `alternative` + `peer_profile` — 来自 `/api/recommend` 实时返回
  - 政策卡 4 张（§6）— 客户端按用户路径从 `data/derived-from-context/policy_basis.json` 静态拉取
  - 故事卡 4 张（§7）— 客户端按 K-NN 匹配结果从 `data/derived-from-context/cases.json` 静态拉取
  - 封面 / 落款 / 风险与边界固定文案 — 客户端模板 hardcode
- 优势：减少 API 数量 + 客户端模板更灵活（不同路径动态组合不同政策/故事卡）

**ER1/ER2/ER3 合规校验**：

- §6 政策卡：用"可能性和方向"措辞，禁用确定因果句式（ER2）
- §7 故事卡：脱敏到地级市 + 单位子类，统一标"X 同学"，不带真名（ER3）
- 整体落款：不带 ww 个人署名，强化"研知道作为机构"的承诺主体（业务必读 §5.1 圈子重合保护）
- 不含任何分享提示 / 公众号关注 / 催单话术（ER3 三个硬约束）

**埋点**：`zexiao_view`（曝光）/ `zexiao_save`（存到相册）

### 7.1 数据反哺后端规则迭代
**两层方案**：

**Layer 1 · 实时采集（前端 → 后端）**

- 每次答题、DP 应答、CTA 点击、择校方向建议生成，全量埋点到后端
- session_id 使用 openid 哈希后的随机串（不可逆），不存原始 openid
- 答题数据脱敏后入数据仓：city → 地级市、单位 → 子类、年龄 → 5 岁段
- 后续 24 小时内的关键转化行为（咨询 / 留资 / 加群 / 加企微）回写到同一 session_id，形成"答题画像 → 后续行为"的全链路样本

**Layer 2 · 周期分析 + 规则迭代（后端 → 06 规则表）**

- **月度对账**（每月第 1 个工作日）：
  - ww 拉一份"上月推荐分布 vs 真实转化率"报表
  - 异常清单：哪些画像组合推荐了 X 路径，但用户后续转化到 Y 路径 / 没转化
  - 对账后产出"规则候选调整清单"，进入 `06-推荐逻辑规则表.md` 的 PROPOSAL 区
- **A/B 测试**（按需）：
  - 同一画像组合，50% 走老规则 50% 走新规则
  - 观察 7 天 / 14 天的咨询率、择校方向建议保存率
  - 实验编号写到 06 的 `experiment_id` 字段，便于追溯
- **季度复盘**（每季末）：
  - 总样本量 ≥ 200 才入正式规则；不足的进灰名单观察期

**反哺数据契约**：

## 八、技术要求概要

### 前端

- 框架：微信小程序原生 或 uni-app（H5+小程序双端）
- 推荐逻辑**不放前端**，所有计算走后端 API
