# V5 结果页数据冲突与匹配规则清单

> 更新时间：2026-05-22  
> 范围：`miniapp/src/pages/result/index.vue`、`miniapp/src/data/real-insights.ts`、推荐策略/案例库/过考率展示口径  
> 目的：记录 V5 结果页联调中发现的“页面已对齐，但数据匹配或展示口径仍不稳”的问题，后续逐条修正。

## 1. 当前已确认的问题

### R-01 相似个案与用户画像不匹配

- 触发路径：
  `党政机关 -> 党员 -> 四川 -> 全日制本科 -> 31-35 -> 本单位晋升 -> 财政发改 -> 2-3万 -> 英语不好 + 数学不好 -> DP001: 没测试过 -> 先有证再优化 -> 说不准 -> 背诵努力`
- 当前结果：
  结果页 `5a · 相似个案` 命中为“卫生系统学员 · 基础弱但时间可投入”，案例画像里出现“未上高中”“卫生系统”等信息。
- 预期结果：
  至少应优先命中“党政机关 / 体制内 / 本科 / 本单位晋升”方向的案例；如果案例库没有完全匹配案例，也不应落到明显跨系统、跨学历层级的案例。
- 当前判断：
  现有 `similarCase` 只按 `studyTime / schoolPreference / mathBase` 做粗匹配，没有纳入 `system / education / goal / primaryPath`。
- 建议处理：
  把 `similarCase` 从“前端临时 if/else”升级为基于案例标签或结构化字段的加权匹配，最低纳入：
  `system`、`goal`、`education`、`primaryPath`、`mathBase`、`studyTime`。

### R-02 相似个案与同画像聚合结论冲突

- 当前结果：
  `5a · 相似个案` 展示“选了：MPA，好上岸”；
  同页 `5b · 同样画像的同学怎么选` 显示“9 位选党校 / 3 位选 MPA”；
  主推荐也是党校。
- 风险：
  用户会感觉页面内部在打架：上面主推党校，中间聚合大多数也选党校，但个案却给了 MPA。
- 建议处理：
  当 `peerDistribution` 明显偏向某一路径时，`5a` 个案应优先从同一路径案例池中挑选；除非明确要表达“少数人为什么改选备选路径”，那就需要额外解释标签。

### R-03 “真实作用”卡片口径仍是固定财政样本

- 当前结果：
  区块 3 第二张卡固定显示“真实作用 · 基于 186 位财政学员数据”。
- 风险：
  对党政机关、教育、医疗等其他系统用户来说，这段标题口径过窄，容易让人误解为整页只适用于财政系统。
- 建议处理：
  选一条：
  1. 改成通用口径，如“基于川渝体制内学员样本”。
  2. 按 `system` 动态切换样本标题。
  3. 若暂时没有分系统样本，就保留正文数字，但在标题或脚注里明确“当前先展示财政样本，后续按系统细分”。

## 2. 当前不是阻断，但要继续盯的点

- `peerDistribution` 当前是前端按主路径写死 `9/3` 或 `3/9` 的演示型聚合，不是运行时真实 K-NN 计算结果。
- 备选方案卡目前会优先从正式院校库找同省 `MPA` 兜底，逻辑是合理的，但还不算真正“与推荐策略联动的第二推荐路径”。
- 本地 H5 测试里 `start_assessment / finish_assessment / view_result` 有埋点 warning；这不影响页面展示，但上线前要确认正式环境埋点是否正常。

## 3. 建议的修复顺序

1. 先修 `R-01` 和 `R-02`：
   相似个案的匹配和路径一致性会直接影响结果页可信度。
2. 再修 `R-03`：
   这更偏文案/样本口径统一，但也会影响“是不是只在讲财政系统”的理解。
3. 最后再决定是否把 `peerDistribution` 从静态演示值接成真实聚合数据。

## 4. 本轮联调结论

- 页面链路：
  `测一测入口 -> 9题 -> Q7 动态岗位 -> Q9 多选 -> DP001 -> 加载页 -> 结果页`
  已完整跑通。
- 当前主要问题已从“页面能不能跑”切换到“数据是不是讲得通”。
