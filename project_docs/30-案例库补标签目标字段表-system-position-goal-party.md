# 案例库补标签目标字段表（system / position / goal / party_member）

> 更新时间：2026-05-22  
> 范围：`studentCases/standardized/student_cases_l1_anonymized.json`、`studentCases/publish/student_cases_miniapp_publish.json`、结果页同行匹配 `5a/5b`  
> 目的：先把案例库补标签需要新增或固化的目标字段表定下来，尤其覆盖 `system_tag / position_tag / goal_tag / party_member_tag`，后续人工补标、脚本抽取、发布层扩展都按这一版执行。

## 1. 这张表解决什么问题

当前结果页同行匹配已经开始依赖更细的画像字段，但案例发布层仍主要靠：

- `title`
- `profile`
- `target`
- `baseline`
- `tags`

做全文命中。

这会导致：

1. 同行匹配不稳定。
2. 同一类用户只能匹配到“年龄差不多、本科、专业差不多”的案例。
3. `财政发改 / 党政机关 / 本单位晋升 / 党员` 这类高价值画像，当前很难被案例库真正承接。

所以这里先不急着写抽取脚本，先把目标字段表定死。

## 2. 建议新增或固化的目标字段

建议在案例标准化层至少明确以下 4 个字段：

1. `system_tag`
2. `position_tag`
3. `goal_tag`
4. `party_member_tag`

建议它们都先进入：

- L1 标准化层：结构化字段
- L2 发布层：可直接被前台读取的字段

`tags[]` 仍然保留，但不再作为唯一承载方式。

## 3. 目标字段总表

| 字段 | 类型 | 是否多值 | 建议层级 | 主要用途 | 当前状态 |
| --- | --- | --- | --- | --- | --- |
| `system_tag` | `string` | 否 | L1 + L2 | 区分用户所在系统，支撑结果页同行匹配 | 缺失 |
| `position_tag` | `string` | 否 | L1 + L2 | 区分岗位方向，支撑“和你同岗位的人怎么选” | 缺失 |
| `goal_tag` | `string[]` | 是 | L1 + L2 | 区分晋升/遴选/防御/职称/转编等目标 | 缺失 |
| `party_member_tag` | `string` | 否 | L1 + L2 | 党校路径下区分党员资格与路径成立条件 | 缺失 |

## 4. 四个字段的标准口径

### 4.1 `system_tag`

#### 标准值

只允许落下面这些值之一：

- `党政机关`
- `公检法纪检`
- `教育系统`
- `医疗系统`
- `乡镇街道`
- `其他事业编`
- `国企央企`
- `银行金融`
- `其他`
- `待确认`

#### 为什么这样定

- 这组值要尽量贴近测一测 Q1，保证结果页画像和案例画像能直接对接。
- 同时保留 `银行金融`，因为现有管综案例里已经出现 `金融监管 / 人民银行` 一类工作语义。

#### 落库规则

- L1：落 `system_tag`
- L2：直接透出 `system_tag`
- `tags[]`：同步写入同名标签，方便旧页面兼容

#### 空值规则

- 不允许空字符串
- 无法判断时统一写 `待确认`

### 4.2 `position_tag`

#### 标准值

先以结果页和测一测当前真的会用到的岗位方向为第一期口径：

- `组织部`
- `宣传部`
- `纪委监委`
- `财政发改`
- `人社民政`
- `应急管理`
- `司法法治信访`
- `综合岗`
- `公安系统`
- `法院检察院`
- `司法行政`
- `教师`
- `教育管理`
- `教育局机关`
- `医护`
- `医院党务行政`
- `卫健委机关`
- `管理岗`
- `技术岗`
- `党建纪检`
- `业务岗`
- `社区村镇管理`
- `社会事务民政`
- `党建组织`
- `监管服务`
- `街道办`
- `金融监管`
- `其他`
- `待确认`

#### 为什么这样定

- 一期不追求无限细分，先覆盖结果页同行匹配当前最需要的岗位方向。
- 保留少量来自现有管综 L1 的高频真实岗位词，如 `监管服务 / 街道办 / 金融监管`，避免把现有信息强行抹平。

#### 落库规则

- L1：落 `position_tag`
- L2：直接透出 `position_tag`
- `tags[]`：同步写入同名标签

#### 空值规则

- 无法稳定归并时，写 `待确认`
- 信息过于宽泛但确实在体制内行政综合岗位时，可写 `综合岗`

### 4.3 `goal_tag`

#### 标准值

建议允许多值，第一期标准值：

- `晋升`
- `遴选`
- `防御`
- `职称`
- `转编`
- `学历提升`
- `自我提升`
- `待确认`

#### 为什么允许多值

- 同一个案例常常同时有两个目标，比如“学历提升 + 遴选”“晋升 + 防御”。
- 如果只允许单值，会损失结果页同行匹配最重要的意图信息。

#### 落库规则

- L1：落 `goal_tag: string[]`
- L2：直接透出 `goal_tag`
- `tags[]`：同步写入每个值

#### 空值规则

- 无法判断时写 `['待确认']`
- 不能用空数组代替

### 4.4 `party_member_tag`

#### 标准值

- `党员`
- `预备党员`
- `非党员`
- `待确认`

#### 为什么必须单独做

- 这是党校路径成立条件的重要字段，不能继续只靠结果页用户答案推断。
- 尤其区分 `四川党校` 和 `重庆党校` 路径时，这个字段很关键。

#### 落库规则

- L1：落 `party_member_tag`
- L2：直接透出 `party_member_tag`
- `tags[]`：同步写入

#### 空值规则

- 当前源表没有时，一律写 `待确认`
- 不允许因为“报了四川党校”就自动推断为 `党员`

## 5. 管综系与党校系的建议取值来源

这是这次最需要区分开的地方。

### 5.1 管综系

现有 L1 标准化层可用字段相对更多，主要来源：

| 目标字段 | 可优先参考的现有 L1 字段 |
| --- | --- |
| `system_tag` | `profile.work_function`、`target.motivation` |
| `position_tag` | `profile.work_function` |
| `goal_tag` | `target.motivation` |
| `party_member_tag` | 当前无稳定字段，需要补录或留空为 `待确认` |

#### 管综系现实判断

- `profile.work_function` 已经有不少真实岗位语义，如 `街道办`、`金融监管`、`交通管理`、`监管服务`
- `target.motivation` 已经有不少目标语义，如“为遴选做准备”“提升学历”“提升发展空间”
- 所以管综系更适合先做“规则抽取 + 人工抽样校验”

### 5.2 党校系

现有 L1 标准化层可用字段明显更少，主要来源：

| 目标字段 | 可优先参考的现有 L1 字段 |
| --- | --- |
| `system_tag` | `profile.work_unit_type` |
| `position_tag` | 当前无稳定字段 |
| `goal_tag` | 当前无稳定字段 |
| `party_member_tag` | 当前无稳定字段 |

#### 党校系现实判断

- 当前党校 L1 普遍只有 `work_unit_type`，而且取值还比较粗，例如：
  - `体制内/机关事业单位`
  - `国企/企业`
  - `单位类型待确认`
- 当前党校 L1 几乎没有岗位方向字段，也没有动机字段。
- 所以党校系不能只靠清洗规则解决，必须接受“源表补录 / 人工补标”这一环。

## 6. 一期映射建议

下面是建议先固定下来的映射口径。

### 6.1 `system_tag` 映射建议

| 来源文本示例 | 目标值 |
| --- | --- |
| `体制内/机关事业单位` | `党政机关` |
| `国企/企业` | `国企央企` |
| `人民银行工作`、`金融监管` | `银行金融` |
| `街道办` | `乡镇街道` |
| `教师`、`学校`、`教育` | `教育系统` |
| `医院`、`医护`、`卫健` | `医疗系统` |
| `法院`、`检察院`、`公安`、`司法`、`纪检` | `公检法纪检` |
| 无法判断 | `待确认` |

说明：

- `体制内/机关事业单位` 先粗归到 `党政机关`，这是一期妥协口径。
- 后续如果党校源表能补更细系统类型，再细分出去。

### 6.2 `position_tag` 映射建议

| 来源文本示例 | 目标值 |
| --- | --- |
| `金融监管` | `财政发改` |
| `监管服务` | `综合岗` |
| `街道办` | `社区村镇管理` |
| `交通管理` | `综合岗` |
| 含 `纪检`、`党建` | `党建纪检` |
| 含 `教师` | `教师` |
| 含 `医院行政`、`党务` | `医院党务行政` |
| 无法判断 | `待确认` |

说明：

- 这一步不要求完美，但要把结果页会用到的典型岗位方向先映射上。
- 对强行映射风险较高的值，宁可写 `待确认`，不要乱归。

### 6.3 `goal_tag` 映射建议

| 来源文本示例 | 目标值 |
| --- | --- |
| `为遴选做准备` | `['遴选', '学历提升']` |
| `调任晋升`、`晋升` | `['晋升', '学历提升']` |
| `提升学历` | `['学历提升']` |
| `提升发展空间` | `['学历提升']` |
| `充实自我`、`提高学习能力` | `['自我提升']` |
| `不想掉队`、`避免被卡` | `['防御', '学历提升']` |
| 无法判断 | `['待确认']` |

说明：

- `goal_tag` 一期应该优先从 `target.motivation` 抽。
- 党校系如果源表没有动机字段，就要人工补标，不建议瞎猜。

### 6.4 `party_member_tag` 映射建议

| 来源 | 目标值 |
| --- | --- |
| 明确写党员 | `党员` |
| 明确写预备党员 | `预备党员` |
| 明确写非党员/群众 | `非党员` |
| 源表未提供 | `待确认` |

说明：

- 这一字段不建议从学校或项目反推。
- 特别是四川党校案例，不能因为“能报”就默认标 `党员`。

## 7. L1 / L2 推荐结构

### 7.1 L1 建议增加

```json
{
  "profile": {
    "system_tag": "党政机关",
    "position_tag": "财政发改",
    "party_member_tag": "党员"
  },
  "target": {
    "goal_tag": ["晋升", "学历提升"]
  }
}
```

### 7.2 L2 建议增加

```json
{
  "system_tag": "党政机关",
  "position_tag": "财政发改",
  "goal_tag": ["晋升", "学历提升"],
  "party_member_tag": "党员"
}
```

这样前台结果页就不需要再只靠 `tags[]` 或正文全文去猜。

## 8. 字段验收标准

### `system_tag`

- 覆盖率目标：一期 `>= 85%`
- `待确认` 占比最好 `<= 15%`

### `position_tag`

- 覆盖率目标：一期 `>= 60%`
- 党校系允许先低一些，但不能长期停留在 `0`

### `goal_tag`

- 覆盖率目标：一期 `>= 70%`
- 至少 `晋升 / 遴选 / 学历提升` 三类要先铺开

### `party_member_tag`

- 覆盖率目标：一期不追求高覆盖
- 但要求所有空缺都明确写 `待确认`
- 禁止隐式默认

## 9. 建议执行顺序

1. 先改标准化和发布层字段定义
2. 先跑管综系规则抽取
3. 再单独整理党校系补录口径
4. 最后再让前台同行匹配优先读结构化字段

## 10. 本轮结论

- `system_tag / position_tag / goal_tag / party_member_tag` 这 4 个字段足够作为案例库补标签第一阶段目标字段表。
- 管综系可先靠现有 L1 字段规则抽取一部分。
- 党校系不能只靠规则，需要接受“源表补录 + 人工补标”这一步。
