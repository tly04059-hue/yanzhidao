# 数据飞轮目录职责与数据结构基线

> 更新时间：2026-05-14  
> 目标：先明确目录边界、数据层级和第一阶段可交付内容，再进入具体实现。  
> 原则：`schooltool` 只读不改；小程序上线链路优先稳定；数据更新必须可追溯、可审核、可回滚。

## 1. 当前目录职责

| 目录 | 职责 | 可写策略 |
| --- | --- | --- |
| `miniapp/` | 小程序/H5 客户端。承载院校库、案例、路径对比、测一测、备考时间线、资料领取。 | 可改，但每次改动必须保证 H5/小程序构建与核心页面可用。 |
| `schoolData/` | 院校库主目录。保存院校源数据、清洗后数据、党校数据、院校库 Web 展示页。 | 可改。新增数据应区分 `sourceData`、清洗中间层、前台发布层。 |
| `studentCases/` | 学生案例库目录。保存历史报名学员、上岸反馈、案例源数据和生成后的标准案例。 | 可改。原始数据保留在 `sourceData`，生成数据另存。 |
| `courseRecommendationStrategy/` | 推荐策略方案库目录。保存提前生成的 N 个策略方案，供测一测按相似度匹配。 | 可改。下一步重点建设。 |
| `AIAgent/` | 数据驱动智能增长飞轮工作区。负责数据清洗、策略生成、AI 检索/问答方案、脚本和中间产物。 | 可改。不要承载小程序运行时代码。 |
| `project_docs/` | 项目推进记录、架构说明、字段标准、更新流程。 | 可改。每个关键阶段应补记录。 |
| `static/` | 静态资料暂存。 | 暂不处理。 |
| `schooltool/` | 老板需求、历史方案、后端旧链路等参考资料。 | 只读不改。 |

## 2. 数据分层

数据后续统一按 4 层理解，避免“信息库”和“前台展示”互相污染。

| 层级 | 说明 | 示例 |
| --- | --- | --- |
| L0 源数据层 | 原始 Excel、CSV、Word、网页抓取结果，只追加不覆盖。 | `schoolData/sourceData/199exam/全量院校表_研招网主表补全.xlsx` |
| L1 标准化层 | 清洗后的结构化数据，字段稳定，可被脚本复用。 | `school_program_master.json`、`student_case_master.json` |
| L2 发布层 | 小程序可直接读取的数据，只包含已审核、完整度达标的字段。 | `miniapp_school_publish.json`、`strategy_publish.json` |
| L3 运营洞察层 | 根据用户行为、留资、反馈生成的分析结果和更新建议。 | 热门院校、FAQ 建议、策略权重调整建议 |

## 3. 三类客户对应内容

### 3.1 还在犹豫是否读研

需要小程序提供：

- 院校库核心字段
- 按相似背景提取的学员案例
- 四个浏览维度：
  - 机关单位系统
  - 目标：晋升、遴选、转岗等
  - 最高学历：大专、本科、研究生
  - 年龄：25-30、30-35、35+
- 路径对比：
  - 管综系：MPA、MBA、MEM
  - 党校方向
  - 双证/单证、是否联考、学费、学制、上课时间、体制内认可、社会认可、考试难度、综合取舍

### 3.2 确定读研但不知道考什么

需要小程序提供：

- 8 题测评
- 用户画像提取
- 与推荐策略方案库做相似度匹配
- 展示最匹配的策略方案：
  - 推荐路径
  - 推荐院校范围
  - 风险提示
  - 本周最小行动计划
  - 资料领取入口

注意：测一测不是每次实时生成全新策略，而是匹配提前生成的 N 个策略方案。

### 3.3 已明确考什么

需要小程序提供：

- 管综系或党校方向备考时间线
- 每个阶段先做什么、再做什么、最后做什么
- 每个时间节点的目标和核心事件
- 免费资料领取入口

## 4. 第一阶段字段标准

### 4.1 院校库一期前台核心字段

一期字段要保证完整度超过 80%，不追求字段多。

| 字段 | 说明 | 是否一期必需 |
| --- | --- | --- |
| `school_id` | 稳定学校 ID | 是 |
| `program_id` | 稳定项目 ID | 是 |
| `school_name` | 学校名称 | 是 |
| `program_type` | MPA/MBA/MEM/党校 | 是 |
| `province` | 省份/地区 | 是 |
| `city` | 城市 | 是 |
| `tuition_min` / `tuition_max` | 学费，单位元 | 是 |
| `duration` | 学制 | 是 |
| `study_mode` | 全日制/非全日制 | 是 |
| `class_time` | 上课方式 | 是 |
| `latest_score` | 最新分数线 | 是 |
| `school_level` | 985/211/双一流/普通/党校 | 建议 |
| `description` | 学校或项目简介 | 建议 |
| `admission_analysis` | 录取分析 | 建议 |
| `exam_subjects` | 考试科目 | 二期 |
| `enrollment` | 招生人数 | 二期 |
| `adjustment` | 是否调剂 | 二期 |
| `interview` | 是否提前面试 | 二期 |

### 4.2 学生案例一期字段

| 字段 | 说明 |
| --- | --- |
| `case_id` | 稳定案例 ID |
| `source_id` | 来源行号或来源文件定位 |
| `age_band` | 年龄段 |
| `system_type` | 单位系统 |
| `education` | 最高学历 |
| `english_base` | 英语基础 |
| `math_base` | 数学基础 |
| `goal` | 读研目标 |
| `budget_band` | 预算区间 |
| `target_path` | 管综/党校/具体项目 |
| `selected_school` | 最终意向或报名院校 |
| `result_status` | 咨询/报名/备考/上岸/未知 |
| `case_summary` | 前台可展示摘要 |
| `match_tags` | 用于匹配用户画像的标签 |

### 4.3 推荐策略方案一期字段

| 字段 | 说明 |
| --- | --- |
| `strategy_id` | 稳定策略 ID |
| `title` | 策略标题 |
| `persona_tags` | 适配用户画像标签 |
| `path_type` | MPA/MBA/MEM/党校/混合对比 |
| `recommended_scope` | 推荐范围，不一定是单所院校 |
| `priority_rules` | 筛选优先级 |
| `risk_warnings` | 风险提示 |
| `weekly_plan` | 本周最小行动 |
| `similar_case_ids` | 可关联案例 |
| `lead_magnet` | 对应免费资料 |
| `version` | 策略版本 |
| `status` | draft/reviewed/published/archived |

## 5. 更新节奏

每年至少 3 次固定更新：

- 3 月：招生简章、报名条件、招生计划、学费、学习方式
- 10 月：报名数据、备考节点、咨询问题、热门路径
- 12 月：考试后反馈、分数线预期、备考转化、案例更新

每次更新影响 3 个库：

- 院校库
- 案例库
- 推荐策略方案库

每次更新需要记录：

- `batch_id`
- `source_files`
- `changed_fields`
- `affected_pages`
- `review_status`
- `published_at`

## 6. 下一步建议

确认本基线后，下一步做：

1. 在 `schoolData/` 下生成院校库标准化文件结构草案。
2. 在 `studentCases/` 下生成案例库标准化文件结构草案。
3. 在 `courseRecommendationStrategy/` 下生成推荐策略方案模板。
4. 先用少量真实数据生成 3-5 个策略方案样例。
5. 再改 `miniapp` 的测一测逻辑，从实时推荐改成策略方案匹配。
