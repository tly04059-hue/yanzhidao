# 案例库标签缺口清单与结果页同行匹配依赖

> 更新时间：2026-05-22  
> 范围：`miniapp/src/data/student-cases-publish.json`、`miniapp/src/data/peer-insights.ts`、结果页 `5a/5b` 同行匹配  
> 目的：把“结果页同行匹配真正依赖的标签”和“当前案例库还缺哪些标签”拆开列清楚，后续补数据或改规则时按同一口径推进。

## 1. 当前结果页同行匹配实际在用什么

结果页当前的同行匹配逻辑在 [peer-insights.ts](/Users/march/yanzhidao/miniapp/src/data/peer-insights.ts:1)。

### 1.1 已实际参与匹配或分布计算的维度

1. `program_type`
   - 用途：先分路径池，决定是在 `党校 / MPA / MBA / MEM` 哪一组里匹配。
   - 当前重要性：最高。

2. `age`
   - 用途：测一测里的 `31-35 / 36-40 / 41+` 会先归并到案例库当前口径 `30-35 / 35+`。
   - 当前重要性：高。

3. `education`
   - 用途：当前只粗分到 `本科 / 大专`。
   - 当前重要性：中高。

4. `goal`
   - 用途：结果页当前把目标映射为 `晋升 / 遴选 / 学历提升` 等标签。
   - 当前重要性：高。

5. `math_base`
   - 用途：映射到 `数学基础弱 / 数学中下 / 数学基础可用`。
   - 当前重要性：中。

6. `english_base`
   - 用途：映射到 `英语待提升 / 英语基础弱 / 英语有基础`。
   - 当前重要性：中。

7. `study_time`
   - 用途：当前只粗映射到 `时间适中 / 时间充足`。
   - 当前重要性：中。

8. `position`
   - 用途：结果页会直接拿岗位词去全文检索，例如 `财政发改`。
   - 当前重要性：高，但当前数据几乎缺失。

9. `system`
   - 用途：结果页会区分 `体制内 / 国企央企` 等系统。
   - 当前重要性：高，但当前数据口径不细。

10. `province`
    - 用途：党校路径下用于区分 `四川党校 / 重庆党校`。
    - 当前重要性：中高。

11. `party_member`
    - 用途：党校路径下会加权。
    - 当前重要性：中高，但当前案例库没有对应标签。

12. `budget`
    - 用途：当前只参与路径分布粗加权，不直接决定个案内容。
    - 当前重要性：中低。

### 1.2 当前匹配方式的现实情况

- 当前不是严格结构化匹配，而是把 `title / profile / target / baseline / tags` 拼成一段文本后做命中判断。
- 也就是说，哪怕某条信息没有单独标签，只要文字里出现，也可能被“碰巧匹配到”。
- 这会带来两个问题：
  1. 没有结构化标签时，匹配稳定性差。
  2. 标题和正文里没有写到的真实信息，就完全匹配不到。

## 2. 当前发布层案例库已有的标签能力

依据当前发布层 [student-cases-publish.json](/Users/march/yanzhidao/miniapp/src/data/student-cases-publish.json:1) 抽样和统计：

- 总案例数：`60`
- 党校案例：`42`
- MPA 案例：`15`
- MBA 案例：`3`

### 2.1 当前全量高频标签

高频标签主要集中在：

- 路径类：`党校系`、`管综系`、`MPA`、`MBA`、`四川党校`
- 年龄类：`25-30`、`30-35`、`35+`
- 学历类：`本科`
- 学科类：`政治学`、`经济学`
- 基础类：`数学基础弱`、`数学中下`、`数学基础可用`、`英语待提升`、`英语有基础`
- 时间类：`时间充足`、`时间适中`
- 目标类：`学历提升`、`晋升`、`遴选`

### 2.2 管综系标签现状

`MPA` 案例已经有一些对结果页同行匹配有帮助的标签：

- `体制内`
- `国企`
- `晋升`
- `遴选`
- 数学/英语基础
- 时间投入
- 首次/非首次备考

这说明管综系案例虽然也不算完美，但已经比党校系更接近“可用来做同行匹配”。

### 2.3 党校系标签现状

党校案例目前的高频标签几乎只有：

- `党校系`
- `四川党校`
- `25-30 / 30-35 / 35+`
- `本科`
- `政治学 / 经济学 / 法学`
- `已录取`
- `已参考`

也就是说，党校案例现在更像“按学校和专业做展示”的数据，不像“按画像做同行匹配”的数据。

## 3. 当前缺的不是哪些字，而是哪些可计算标签

下面这些是结果页同行匹配真正需要、但当前发布层明显缺口较大的标签。

### G-01 系统标签缺失

当前测一测系统选项：

- `党政机关`
- `公检法纪检`
- `教育系统`
- `医疗系统`
- `乡镇街道`
- `其他事业编`
- `国企央企`

当前案例库里能稳定命中的只有：

- `体制内`
- `国企`

明显缺失：

- `党政机关`
- `公检法纪检`
- `教育系统`
- `医疗系统`
- `乡镇街道`
- `其他事业编`

影响：

- 结果页只能知道“这是体制内”，但不知道“是哪一种体制内”。
- 所以 `财政发改` 这样的党政机关用户，最终只能匹配到“30-35 本科 政治学”的党校案例，像不像本人只能靠运气。

### G-02 岗位方向标签缺失

结果页已经在用岗位做匹配，例如：

- `财政发改`
- `组织部`
- `宣传部`
- `纪委监委`
- `人社民政`
- `应急管理`
- `司法法治信访`
- `综合岗`

当前案例库中，上述词在发布层里基本是 `0` 命中。

影响：

- `position` 这个高价值字段虽然已经接进匹配规则，但当前几乎不产生实际作用。
- 结果页最想回答的“和你同岗位的人怎么选”，目前还做不到。

### G-03 目标标签不完整

当前测一测目标包括：

- `本单位晋升`
- `遴选`
- `防御`
- `职称`
- `转编`

当前案例库里可用标签只有：

- `晋升`
- `遴选`
- `学历提升`
- 少量 `自我提升`

缺失或严重不足：

- `防御`
- `职称`
- `转编`

影响：

- 当前 `goal` 只能粗归并，不能精细区分“为了单位内部竞争防御”和“为了遴选/职称/转编”。

### G-04 党员标签缺失

党校路径下，`party_member` 已经被结果页加权使用，但案例库当前没有：

- `党员`
- `预备党员`
- `非党员`

影响：

- 党校个案无法体现“这条路径是不是因为党员身份成立”。
- 四川党校和重庆党校的差异，也无法通过案例标签真正表达。

### G-05 党校省份/学校标签不完整

当前党校案例几乎都是：

- `四川党校`

但结果页和推荐策略其实已经需要比较：

- `四川党校`
- `重庆党校`

影响：

- 党校系同行分布可以展示成 `四川党校 / MPA`，但如果后续要更准确比较 `四川党校 vs 重庆党校`，当前案例库支撑不够。

### G-06 学历起点口径过粗

当前党校案例基本都是：

- `本科`

缺少更细口径：

- `全日制本科`
- `非全日制本科`
- `大专起点后升本`
- `专升本`

影响：

- 结果页只能知道“都是本科”，但不能判断用户是不是更像“全日制本科党政机关”还是“非全本科在职提升”。

### G-07 时间投入口径在党校案例中几乎缺失

管综案例已有：

- `时间充足`
- `时间适中`

但党校案例多为：

- `总学习时长 xxxx`

没有统一转换成前台可用标签，例如：

- `时间紧`
- `时间适中`
- `时间充足`

影响：

- 当前 `study_time` 对党校案例几乎不起作用。

### G-08 英语/数学基础标签在党校案例中基本缺失

党校路径虽不考英数，但用户在结果页仍然会因为 `Q9 + DP001` 被区分。

当前党校案例基本没有：

- `英语待提升`
- `英语基础弱`
- `英语有基础`
- `数学基础弱`
- `数学中下`
- `数学基础可用`

影响：

- 结果页虽然能把“英数弱、但更适合党校路径”讲出来，但党校同行个案本身无法承接这种画像。

## 4. 结果页同行匹配最小可用标签集

如果只考虑结果页 `5a/5b`，不是先追求完美案例库，而是先让“同行匹配讲得通”，建议最小可用标签集至少有这些：

1. `program_type`
2. `party_school`
3. `age_bucket`
4. `education_level`
5. `system_tag`
6. `position_tag`
7. `goal_tag`
8. `party_member_tag`
9. `math_base_tag`
10. `english_base_tag`
11. `study_time_tag`

其中最急的是前 8 个，尤其是：

- `system_tag`
- `position_tag`
- `goal_tag`
- `party_member_tag`

因为这 4 个直接决定“像不像这个用户本人”。

## 5. 建议的补标签顺序

### P0 先补

1. `system_tag`
2. `position_tag`
3. `goal_tag`
4. `party_member_tag`
5. `party_school`

原因：

- 这是结果页党校路径最缺、也是最影响可信度的一组。

### P1 再补

6. `education_level`
7. `study_time_tag`
8. `math_base_tag`
9. `english_base_tag`

原因：

- 这些会让相似度更细，但不会像前面几项那样直接造成“完全不像本人”。

### P2 最后统一

10. `result_stage`
11. `attempt_count`
12. `budget_tag`
13. `province_tag`

原因：

- 这些有帮助，但不是当前结果页同行模块的第一阻断点。

## 6. 建议的数据改造方式

不建议继续只靠 `title/profile/baseline/tags` 的全文匹配往前堆。

建议后续在案例发布层增加明确字段，再由前台决定是否回退到标签：

- `system_tag`
- `position_tag`
- `goal_tag`
- `party_member_tag`
- `education_level`
- `study_time_tag`
- `math_base_tag`
- `english_base_tag`

前台结果页的使用顺序建议是：

1. 先读结构化字段
2. 再读标签
3. 最后才回退到标题/正文全文匹配

## 7. 本轮结论

- 当前结果页同行匹配规则已经比之前稳定，至少不会再出现“主推党校，但相似个案是卫生系统 MPA”的明显冲突。
- 但当前案例库，尤其是党校案例库，仍然缺少足够的画像标签。
- 现在的主要矛盾已经不是“有没有案例”，而是“案例能不能被结果页按画像正确挑出来”。
