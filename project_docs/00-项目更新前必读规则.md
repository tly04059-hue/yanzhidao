# 项目更新前必读规则

> 更新时间：2026-05-15  
> 适用范围：`projectOps/`、`miniapp/`、`schoolData/`、`studentCases/`、`courseRecommendationStrategy/`、`AIAgent/`、`project_docs/`  
> 目的：每次开始更新前先统一口径，避免数据清洗、页面展示、策略生成时出现字段错位或颗粒度混乱。

## 1. 目录边界

| 目录 | 规则 |
| --- | --- |
| `schooltool/` | 只读不改。只用于理解老板需求和历史方案。 |
| `projectOps/` | 大项目中控台。只负责调度、同步、校验、构建和发版报告，不承载业务数据、不写页面逻辑。 |
| `miniapp/` | 客户端主项目。任何改动必须保证浏览、交互、构建、提测链路稳定。 |
| `schoolData/` | 院校库主目录。所有院校相关源数据、标准化数据、发布数据都放这里。 |
| `studentCases/` | 学生案例库。后续案例提取、案例标签、案例发布数据放这里。 |
| `courseRecommendationStrategy/` | 推荐策略方案库。测一测匹配的是这里提前生成的策略方案。 |
| `AIAgent/` | 数据飞轮、清洗脚本、AI 检索/问答方案、中间分析产物。 |
| `project_docs/` | 项目规则、阶段记录、字段标准、更新说明。关键变更必须留档。 |

## 2. 数据分层

| 层级 | 规则 |
| --- | --- |
| L0 源数据层 | 原始 Excel、CSV、Word、网页抓取结果只追加不覆盖，保留来源。 |
| L1 标准化层 | 清洗后的完整信息库，字段稳定，服务小程序、策略库、AI 问答和数据飞轮。 |
| L2 发布层 | 前台可直接读取的数据，只放已审核、完整度达标、适合展示的字段。 |
| L3 洞察层 | 用户行为、留资、反馈、热门院校、FAQ、权重调整建议。 |

## 3. 前台展示和标准化 JSON 的铁规则

前台可以展示合并文本，但标准化 JSON 必须同时保留拆分字段和展示字段。

例如：

```json
{
  "school_code": "10610",
  "school_name": "四川大学",
  "school_label": "(10610)四川大学"
}
```

同样规则适用于：

| 对象 | 标准化 JSON 必须保留 | 前台可展示 |
| --- | --- | --- |
| 学校 | `school_code`、`school_name`、`school_label` | `school_label` 或 `school_name` |
| 院系 | `department_code`、`department_name`、`department_label` | `department_label` |
| 专业类别 | `major_category_code`、`major_category_name`、`major_category_label` | `major_category_label` |
| 专业 | `major_code`、`major_name`、`major_label` | `major_label` |

禁止为了前台方便，只保留合并文本。否则后续去重、更新、对比、推荐策略和 AI 检索都会出问题。

## 4. 管综院校库颗粒度

管综系 MPA、MEM、MBA 的标准化最小颗粒度不是“学校”，而是：

```text
项目 -> 学校 -> 院系 -> 专业类别 -> 专业 -> 研究方向 -> 学习方式
```

因此一所学校可能生成多条标准化记录。比如四川大学 MEM 下不同专业或方向，应拆成不同 `record_id`。

## 5. 管综字段表的展示规则

`schoolData/sourceData/199exam/管综院校字段表.xlsx` 是管综院校库字段标准之一。

已确认规则：

- A-Z 列是一阶段字段，也是小程序全年展示字段，必须优先补齐。
- 展示顺序按 `MPA -> MEM -> MBA`。
- 筛选维度优先级为：`项目` 第一，`所在省` 第二，`所在市` 第三。
- `研究方向` 不写“全日制”或“非全日制”，学习方式单独放入 `study_mode`。
- 管综前台院校性质只展示 `985`、`211`、`双非` 三类；标准化层可以保留更丰富标签。
- 如果调剂录取分析里的 `名额` 是正数，`是否调剂` 应推导为 `是`。

## 6. 小程序一期展示口径

小程序一期只展示核心字段，不把信息库所有字段都直接放到前台。

一期优先展示：

- 招生专业
- 专业培养方向
- 招生总人数
- 各专业人数
- 上线人数
- 报考录取率
- 剔除弃考实考录取率
- 学费
- 上课地点及上课方式
- 最新分数线
- 是否接受大专报考
- 是否调剂
- 退役计划
- 少骨计划
- 必要备注

历史数据、复试考情、录取分析可以进入信息库；前台只展示最新且经过整理的信息。

## 7. 年度更新规则

每年至少三次固定更新：

- 3 月：招生简章、招生计划、学费、学习方式、分数线。
- 10 月：报名数据、热门路径、咨询问题、备考节点。
- 12 月：考试反馈、案例更新、分数线预期、策略复盘。

每次更新要同时检查：

- 院校库是否变化。
- 案例库是否变化。
- 推荐策略方案库是否需要同步更新。
- 小程序展示字段是否仍然完整度达标。

## 8. projectOps 中控台规则

跨目录的数据更新、同步、校验和构建，优先通过 `projectOps/` 执行，不再靠人工手工搬运发布文件。

最小流水线：

```bash
python3 projectOps/scripts/update-all.py
python3 projectOps/scripts/sync-miniapp-data.py
python3 projectOps/scripts/check-all.py
python3 projectOps/scripts/build-miniapp.py --target h5
```

一键提测/发版前检查：

```bash
python3 projectOps/scripts/release.py --target h5
```

微信小程序构建可使用：

```bash
python3 projectOps/scripts/release.py --target mp-weixin
```

`projectOps/` 只调度现有脚本：

- 院校库仍由 `schoolData/` 保存源数据、标准化数据和发布数据。
- 案例库仍由 `studentCases/` 保存源数据、匿名标准化数据和发布数据。
- 推荐方案库仍由 `courseRecommendationStrategy/` 保存策略方案。
- 小程序仍只负责读取 `miniapp/src/data/` 中的发布数据并展示。
- 每次运行中控脚本后，报告写入 `projectOps/reports/`。

## 9. 开工前检查清单

每次更新或写代码前，先检查：

1. 是否会修改 `schooltool/`；如果会，停止，除非用户明确要求。
2. 当前改动属于 L0、L1、L2 还是 L3。
3. 是否破坏了 `code/name/label` 同时保留的规则。
4. 是否把历史深度数据误放到了前台展示层。
5. 是否影响小程序提测、部署、上线链路。
6. 是否需要同步更新 `project_docs/`。
7. 是否属于跨目录更新；如果是，优先走 `projectOps/`。
8. 是否需要同步更新推荐策略方案库或案例库。

## 10. 当前一期上线任务边界

> 更新时间：2026-05-15  
> 这部分记录用户已确认的本版取舍。后续任务开始前必须先看，避免反复提示已延后的事项。

一期上线前继续推进：

1. 按 `schooltool/docs/ui_samples` 的 HTML 样稿统一小程序视觉，重点先对齐院校库、详情页、了解页和测一测链路。
2. 推进正式 API 和域名配置。
3. 打通留资闭环。
4. 完成微信开发者工具真机测试。
5. 推荐方案库是本版重点，等党校详细内容补齐后，需要继续细化策略方案。
6. 党校院校库并入小程序与推荐方案库细化同步处理，等党校详细数据确认后再推进。

党校 V2 数据已开始接入时，还必须先读：

- `project_docs/24-党校V2数据清洗与小程序接入口径.md`

党校数据更新时，不能只看前台字段。FAQ、特殊政策、川渝党校对比、历年报录和分数线都要保留在 L1 信息库或 `dx_data.json` 分板块里，L2 小程序发布层只放审核后的核心展示字段。

一期上线前暂不推进：

1. 估分工具正式化。当前保留占位，等后续有内容后再增加。
2. 案例库筛选增强。本版不再更新案例筛选能力，放到第二版或第三版。
3. MEM 字段补齐。本版暂不补，放到第二版或第三版。

注意：

- 不要在后续上线前缺口清单里反复把“估分工具正式化”“案例库筛选增强”“MEM 字段补齐”列为本版必须完成项。
- 推荐方案库、党校院校库仍是本版重要任务，但依赖同事补齐党校详细资料。
