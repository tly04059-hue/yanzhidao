# 管综院校字段表解读与落库规则

> 更新时间：2026-05-14  
> 来源文件：`schoolData/sourceData/199exam/管综院校字段表.xlsx`  
> 适用范围：MPA、MEM、MBA 管综院校库标准化、发布展示、推荐策略生成。

## 1. 文件定位

这份 Excel 不是全量院校数据表，而是管综院校库的字段标准和分阶段采集模板。

它的作用是：

- 明确小程序一期展示哪些字段。
- 明确后续二期、三期要补充哪些深度字段。
- 明确 MPA、MEM、MBA 的展示顺序和筛选优先级。
- 明确学校、院系、专业类别、专业的编码和名称处理规则。

## 2. 分阶段字段

| 阶段 | 字段范围 | 用途 |
| --- | --- | --- |
| 一阶段 | A-Z 列 | 小程序全年展示字段，放最新数据，完整度优先。 |
| 二阶段 | 复试考情年度字段 | 保存 2026、2025、2024 等年度复试内容、形式、成绩计算、参考书等。 |
| 三阶段 | 一志愿录取分析、调剂录取分析 | 保存年度录取数据、复录比、录取率、分数段、调剂名额和轮次等。 |

## 3. 一阶段小程序展示字段

一阶段字段应优先补齐，后续小程序发布层也应围绕这些字段设计。

## 4. code/name/label 保留规则

前台可以展示合并文本，但 L1 标准化层必须同时保留拆分字段和展示字段。

| 对象 | 必须保留 |
| --- | --- |
| 学校 | `school_code`、`school_name`、`school_label` |
| 院系 | `department_code`、`department_name`、`department_label` |
| 专业类别 | `major_category_code`、`major_category_name`、`major_category_label` |
| 专业 | `major_code`、`major_name`、`major_label` |

示例：

```json
{
  "school_code": "10610",
  "school_name": "四川大学",
  "school_label": "(10610)四川大学",
  "department_code": "101",
  "department_name": "经济学院",
  "department_label": "(101)经济学院",
  "major_category_code": "1252",
  "major_category_name": "公共管理",
  "major_category_label": "(1252)公共管理",
  "major_code": "125200",
  "major_name": "公共管理",
  "major_label": "(125200)公共管理"
}
```

## 5. 管综数据颗粒度

标准化最小颗粒度为：

```text
项目 -> 学校 -> 院系 -> 专业类别 -> 专业 -> 研究方向 -> 学习方式
```

因此，同一学校在以下任意字段不同的情况下，都可以拆成不同记录：

- 项目不同：MPA / MEM / MBA
- 院系不同
- 专业类别不同
- 专业不同
- 研究方向不同
- 学习方式不同

## 6. 清洗规则

1. `研究方向` 中不要包含“全日制”“非全日制”等学习方式字样。
2. 学习方式统一放入 `study_mode`。
3. 学费从“万”统一换算为“元”，例如 `8.4` 万应落为 `84000`。
4. 管综发布层院校性质可展示为 `985`、`211`、`双非`；L1 可以保留更丰富标签，如 `双一流`。
5. 如果调剂录取分析里的 `名额` 是正数，则 `adjustment` 应为 `是`。
6. A-Z 一阶段字段应优先计算完整度，完整度不足 80% 的记录原则上不进入一期前台发布层。

## 7. 对推荐策略的价值

这份字段表后续可以支撑以下推荐策略：

- 地区优先型：按省、市筛选。
- 预算敏感型：按学费区间筛选。
- 在职友好型：按非全日制、周末班、集中班、上课地点筛选。
- 稳妥上岸型：按复试线、录取率、复录比、招生人数筛选。
- 调剂机会型：按历史调剂名额、调剂轮次筛选。
- 名校认可型：按 985、211、双一流、体制内认可度筛选。

## 8. 与小程序的关系

小程序一期不直接展示全部历史字段。

推荐做法：

- L1 标准化层保存完整字段。
- L2 发布层只输出适合展示的最新核心字段。
- 测一测和策略库可以读取 L1/L2 的结构化字段，生成或匹配提前准备好的策略方案。
- AI 问答后续可检索 L1 的深度字段，但回答前要转成用户能理解的表达。
